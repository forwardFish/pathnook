## 三、必须修改项（P0）

### P0-1：Freemius 成功回跳地址改成 `/api/checkout`

#### 当前问题
你现在实际测试用的是：

```text
https://www.pathnook.com/dashboard/billing
```

但仓库里真正负责处理 Freemius 回跳、写入 entitlement、再跳转到账单页的是：

```text
app/api/checkout/route.ts
```

如果直接跳 `/dashboard/billing`，就会绕开这条真正的 completion 逻辑。

#### 为什么必须改
`app/api/checkout/route.ts` 已经实现了这些动作：

- 读取 Freemius 回跳 URL
- 解析 completion payload
- 识别用户
- `applyCheckoutCompletionForUser(...)`
- 再跳到 `/dashboard/billing?checkout=success&plan=...`

这条路由本来就是为 Freemius success redirect 准备的。

#### 你现在要改哪里
去 **Freemius Dashboard → Plans → Customization**

把：

```text
https://www.pathnook.com/dashboard/billing
```

改成：

```text
https://www.pathnook.com/api/checkout
```

#### 改完后的效果
Freemius 支付成功后会：

1. 回到 `/api/checkout`
2. Pathnook 服务端完成 entitlement 更新
3. 再重定向到 `/dashboard/billing?checkout=success...`

#### 这一步不需要改代码
只需要改 Freemius Dashboard 里的 success redirect 配置。

---

### P0-2：Freemius webhook 指向 `/webhook`

#### 当前问题
仓库里现成的 webhook 入口是：

```text
app/webhook/route.ts
```

而不是：

```text
app/api/freemius/webhook/route.ts
```

#### 正确配置
Freemius Dashboard 里 webhook endpoint 应该填：

```text
https://www.pathnook.com/webhook
```

#### 不要填
```text
https://www.pathnook.com/api/freemius/webhook
```

因为仓库里没有这个路由。

#### 你现在要做什么
去 **Freemius Dashboard → Webhooks**

确认 listener URL 是：

```text
https://www.pathnook.com/webhook
```

---

### P0-3：生产环境关闭 Creem fallback

#### 当前问题
`.env.example` 里虽然主 provider 已经是 Freemius，但仍然保留了 Creem fallback：

```env
BILLING_PROVIDER_ACTIVE=freemius
PAYMENT_PROVIDER_DEFAULT=freemius
BILLING_PROVIDER_ALLOW_CREEM_FALLBACK=1
FEATURE_CREEM_UI=0
FEATURE_CREEM_ROUTES=1
```

而 `lib/payments/provider-selection.ts` 和 `lib/payments/service.ts` 仍然支持：

- Freemius 不可用时自动回退 Creem
- 兼容读取 Creem provider

这会导致生产环境如果 Freemius 某项环境变量漏配，项目可能静默切回 Creem。

#### 生产环境建议
把生产环境变量改成：

```env
BILLING_PROVIDER_ACTIVE=freemius
PAYMENT_PROVIDER_DEFAULT=freemius
BILLING_PROVIDER_ALLOW_CREEM_FALLBACK=0
FEATURE_CREEM_UI=0
FEATURE_CREEM_ROUTES=0
```

#### 如果你用 Vercel
去：

- Vercel Project
- Settings
- Environment Variables

把上面 4 个变量按这个值覆盖到 Production。

---

### P0-4：Billing 页让 one-time 用户也能看到 portal 按钮

#### 当前问题
`app/(dashboard)/dashboard/billing/page.tsx` 里目前只有 recurring 用户才显示 portal 按钮。

也就是现在的逻辑近似于：

```ts
{hasRecurringPlan ? <Button>Open Freemius Billing Portal</Button> : null}
```

#### 为什么建议改
one-time diagnosis 用户也可能需要：

- 查看发票
- 查看支付记录
- 查看 billing portal
- 处理订单问题

不应该因为不是 recurring 就完全看不到 portal 入口。

#### 建议改法
把显示条件从：

```ts
hasRecurringPlan
```

改成：

```ts
snapshot.portalAvailable
```

#### 目标效果
只要本地 snapshot 判断 portal 可用，就显示 portal 按钮，不再只给 recurring 用户。

## 四、Creem 相关内容怎么处理

你前面问的是：

> 这些都可以注释掉吗？

可以，但我建议分两层处理。

---

### 方案 A：先“禁用”，不立刻删除
如果你怕回滚困难，先这样做：

#### 1）保留文件，但不走 Creem
保留这些文件不删：

- `app/api/creem/webhook/route.ts`
- `lib/payments/creem.ts`
- `lib/payments/providers/creem.ts`（如果存在）
- 其他 `creem` 相关代码

#### 2）环境变量彻底关掉
生产环境用：

```env
BILLING_PROVIDER_ALLOW_CREEM_FALLBACK=0
FEATURE_CREEM_UI=0
FEATURE_CREEM_ROUTES=0
```

#### 3）前端不要再暴露任何 Creem 入口
全站公开页、billing、pricing、help、faq 里不要再出现 Creem。

#### 这个方案的优点
- 风险低
- 好回滚
- 上线快

#### 这个方案的缺点
- 仓库里还留着旧代码
- 技术债还在

---

### 方案 B：确定不用 Creem 后，直接删干净
如果你确认以后都不会再切回 Creem，就建议彻底删。

#### 删除这些文件
优先删或弃用：

- `app/api/creem/webhook/route.ts`
- `lib/payments/creem.ts`
- `lib/payments/providers/creem.ts`（如存在）
- 其他 `creem` 相关实现文件

#### 修改这些文件
##### `lib/payments/provider-selection.ts`
去掉：

- `creemConfigured`
- `allowCreemFallback`
- fallback 到 `creem` 的逻辑

收敛成只返回 `freemius`

##### `lib/payments/service.ts`
去掉：

- `providers.creem`
- `getConfiguredBillingPlans as getConfiguredCreemBillingPlans`
- 默认 providerName = `'creem'`
- `getCompatibilityBillingProvider(name = 'creem')`

统一改成只保留 Freemius。

#### 修改 `package.json`
删掉依赖：

```json
"creem": "^1.4.4",
"creem_io": "^1.1.0"
```

然后重新安装依赖。

#### 修改 `.env.example`
删除这些变量：

```env
CREEM_API_KEY=
CREEM_WEBHOOK_SECRET=
CREEM_TEST_MODE=
CREEM_PRODUCT_ONE_TIME_ID=
CREEM_PRODUCT_MONTHLY_ID=
CREEM_PRODUCT_ANNUAL_ID=
BILLING_PROVIDER_ALLOW_CREEM_FALLBACK=
FEATURE_CREEM_UI=
FEATURE_CREEM_ROUTES=
```

#### 这个方案的优点
- 仓库干净
- 上线形态明确
- 排错简单

#### 这个方案的缺点
- 如果后面想回滚到 Creem，要重新恢复代码
