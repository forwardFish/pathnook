# 需求文档_1.2.md
# Parent Dashboard 专业诊断系统需求文档（在现有系统基础上的增量版本）
**版本**：v1.2  
**文档类型**：增量需求文档 / 可直接交给 Codex 开发  
**目标**：在当前已完成的 Parent Dashboard 系统基础上，补齐“专业诊断能力”这条核心主线，让系统能够稳定生成 **结构化、可解释、可执行、可复盘** 的学习诊断报告。  
**适用阶段**：MVP+ / Beta 前关键增强版本  
**市场定位**：欧美家长（parent-first），数学优先，Web 产品

---

# 0. 文档目的

这份 1.2 文档不是重写整个产品，而是专门解决一个核心问题：

> **如何让系统产出的诊断，真正像“专业诊断书”，而不是 AI 套话报告。**

当前系统已经具备：
- 注册登录
- child profile
- 上传 5–10 页材料
- upload -> run -> report 主流程
- Diagnosis / Evidence / 7-Day Plan 页面骨架
- tutor share
- billing / paywall
- EN/ES 输出
- PDF export
- admin review 基础能力

但当前最关键的商业壁垒还没有完全做深：

> **专业诊断能力本身**

也就是：
- 需要什么输入，才能做出可信判断
- 怎样把图片/PDF 变成结构化 problem items
- 怎样把 problem items 变成 taxonomy labels
- 怎样从 labels 推出 primary diagnosis / secondary diagnosis
- 怎样从 diagnosis 推出 7-Day Plan
- 怎样控制质量与置信度
- 怎样让报告真正像“判断系统”而不是“内容生成”

# 2. 为什么这一层必须单独做成文档

## 2.1 业务原因
用户是否愿意付：
- $19 单次
- $39/月订阅
- $299/年长期系统

核心不在页面有多漂亮，而在这里：

> **这份报告看起来是不是专业、是不是可信、是不是能指导家长本周做决定。**

## 2.2 风险原因
如果不单独把“诊断系统”设计清楚，产品很容易变成：

- AI 看图总结
- 套模板报告
- 空泛建议
- 看不出证据
- 看不出“为什么是这个判断”
- 周报之间高度重复

这些都会直接导致：
- 首单转化差
- 复购差
- 月订阅留存差
- 年费完全卖不动

## 2.3 竞争原因
市场上已经存在“诊断 + next steps”的相邻系统，但大多是：
- 平台内测评诊断
- 教师反馈工具
- 练习题推荐系统

你的差异化必须建立在：

> **基于真实作业 / 测验 / 订正页做 evidence-backed root-cause diagnosis**

而不是只做“总结”。

# 4. 核心设计原则

## 原则 1：先抽事实层，再写报告
严禁模型直接看图片后写长篇报告。  
必须两阶段：

### 第一阶段：事实层
抽取：
- page
- problem
- student response
- teacher mark
- evidence anchor
- taxonomy labels

### 第二阶段：判断层
聚合：
- primary diagnosis
- secondary diagnosis
- pattern issues
- sporadic issues
- plan

## 原则 2：每个判断都必须能回溯到证据
没有 evidence 的 finding 不允许进 Top Findings。

## 原则 3：诊断必须区分“表面错误”和“根因”
不能只说“这题错了”，而要说：
- 这类错反复出现
- 更像 concept / procedure / reading / notation 哪一类问题
- 为什么不是另外几类原因

## 原则 4：处方必须比建议更具体
不能写“建议加强练习”。  
必须写：
- 每天做什么
- 做几分钟
- 家长问什么
- 今天成功标准是什么
- 本周先别做什么

## 原则 5：低置信度时宁愿少说，也不要乱说
必须建立 confidence gate。

### 6.2.2 新增输入项：最近表现趋势（可选但强烈建议）
字段名：`recentTrend`

可选值：
- `improving`
- `stable`
- `declining`
- `not_sure`

作用：
- 帮助判断“偶发”还是“持续问题”
- 帮助报告 summary

### 6.2.3 新增输入项：老师是否有批注（可选）
字段名：`teacherFeedbackPresent`
类型：boolean

作用：

### 6.2.4 新增输入项：家长最担心的问题（可选）

可选值（可多选）：
- `concepts`

- `carelessness`

- `confidence`
- `not_sure`

作用：
- 帮助生成更贴近家长决策的总结

### 6.2.5 新增输入项：是否已有 tutor（可选）

可选值：
- `none`

作用：

### 6.2.6 新增输入项：本次材料类型（现有 sourceType 做规范化）
字段名：`sourceType`
建议统一枚举：
- `homework`
- `quiz`
- `test`
- `corrections`
- `mixed`

## 6.3 上传页面要新增的表单区块
在 `/dashboard/children/[childId]/upload` 页面新增一个 **Diagnostic Intake Card**：

### 区块标题
**Help us make the diagnosis more accurate**

### 字段顺序
1. What are you trying to understand today?
2. How has math been going recently?
3. Is there teacher feedback on these pages?
4. What are you most worried about?
5. Is your child already working with a tutor?
6. Notes (optional)

## 6.4 输入材料准备要求（必须给用户明确说明）
为了生成专业诊断，系统必须在上传页明确要求用户尽量准备：

### 推荐材料清单
- 最近 5–10 页数学作业 / quiz / test
- 尽量包含错题和订正页
- 如果有老师批改痕迹，更优先
- 不要只传全对页面
- 如果是文字题问题，尽量包含完整题面

### 不推荐材料
- 过于模糊的照片
- 裁掉题目上半部分的图片
- 只有答案、没有过程的页面
- 多学科混在同一批里（MVP 阶段）

# 7. Problem Item 抽取层需求

## 7.1 目标
把上传的页面，转成一组结构化 `problem_items`。  
这是整个诊断系统的事实层基础。

## 7.2 每个 problem item 必须输出的字段
最小结构：

```json
{
  "pageNumber": 2,
  "problemNumber": "Q4",
  "questionSummary": "Fraction addition with unlike denominators",
  "studentResponseSummary": "Added numerators directly without finding a common denominator",
  "teacherMark": "wrong",
  "confidence": 0.82
}
```

## 7.3 MVP 阶段的抽取字段定义

### 必填字段
- `pageNumber`
- `problemNumber`
- `questionSummary`
- `studentResponseSummary`
- `teacherMark`
- `confidence`

### 强烈建议同时产出
- `sourceSnippet`
- `evidenceAnchor`
- `workType`（multiple choice / open response / computation / word problem）
- `topicHint`（fractions / equations / geometry 等）

## 7.4 抽取规则
### questionSummary
不是完整 OCR 复制，而是简短摘要。  
目标是让后续模型与页面都能读懂。

### studentResponseSummary
必须强调“学生做了什么”，而不是直接判断对错。

### teacherMark
枚举：
- `correct`
- `wrong`
- `partial`
- `unknown`

### confidence
由 OCR + extraction 一起决定。

## 7.5 需要改动的数据库
现有 `problem_items` 建议新增：

- `questionSummary`
- `studentResponseSummary`
- `workType`
- `topicHint`
- `sourceSnippet`
- `normalizedTeacherSignal`
- `uiAnchorKind`

## 7.6 需要改动的 API
新增或增强：
- `POST /api/runs/[runId]/extract-items`
- `GET /api/runs/[runId]/items`

# 8. Taxonomy 分类层需求

## 8.1 目标
对每个 `problem_item` 进行错误归类，形成结构化错误标签。

## 8.2 MVP taxonomy（固定 8 类）
1. `concept_gap`
2. `procedure_gap`
3. `calculation_slip`
4. `reading_issue`
5. `notation_error`
6. `strategy_error`
7. `careless_slip`
8. `incomplete_reasoning`

## 8.3 每个 item 的输出结构
```json
{
  "errorType": "procedure_gap",
  "severity": "high",
  "confidence": 0.76,
  "rationale": "Student skipped the common denominator step before adding fractions."
}
```

## 8.4 分类规则
### severity
- `low`
- `medium`
- `high`

### confidence
范围 0–1

### rationale
必须是 1–2 句，且尽量基于 evidence，不写空话。

## 8.5 item 可支持多个标签，但 MVP 建议最多 2 个
例如：
- primary label
- secondary label

## 8.6 分类后的持久化
现有 `item_errors` 表继续使用，但必须确保：
- 支持多 label
- 支持 severity
- 支持 rationale
- 支持 confidence

# 9. 诊断层需求（Issue Tree + Hypothesis Scoring）

## 9.1 目标
不是直接“汇总标签”，而是形成真正的诊断结论。

## 9.2 诊断层必须输出的内容
- `primaryDiagnosis`
- `secondaryDiagnosis`
- `patternIssues`
- `sporadicIssues`
- `doNotOverreactPoints`

## 9.3 Issue Tree（学习诊断问题树）
一级问题域：
- Concept
- Procedure
- Reading
- Notation
- Calculation
- Strategy
- Carelessness
- Communication

二级根因可做内部映射，例如：
- `procedure_gap -> missing_step_order`
- `reading_issue -> condition_misread`
- `notation_error -> sign_or_unit_loss`

## 9.4 Hypothesis Scoring 规则
对候选 diagnosis 进行打分：

```text
Diagnosis Score =
(error frequency * 0.4)
+ (severity weight * 0.3)
+ (cross-page repetition * 0.2)
+ (teacher signal support * 0.1)
```

## 9.5 pattern issue 判定
满足任一：
- 同 taxonomy 在 2 页以上出现
- 同 taxonomy 出现 ≥3 次
- 连续两次 report 中都出现

## 9.6 sporadic issue 判定
满足：
- 仅出现 1 次
- 不构成重复模式
- severity 不高

## 9.7 do-not-overreact points
系统必须明确告诉家长：
- 哪些问题现在不用过度反应
- 哪些只是偶发粗心
- 哪些不应该本周成为主重点

## 9.8 数据库存储建议
在 `reports` 或新聚合表中持久化：
- `primaryDiagnosisCode`
- `secondaryDiagnosisCodes[]`
- `patternSummaryJson`
- `sporadicSummaryJson`
- `doNotOverreactJson`

# 10. 处方层需求（7-Day Plan Generator）

## 10.1 目标
把 diagnosis 变成具体可执行的 7 天行动计划。

## 10.2 原则
不能只靠 LLM 自由发挥。  
必须有规则库 + LLM 组合。

## 10.3 每天的输出结构
```json
{
  "day": 1,
  "minutes": 20,
  "goal": "Rebuild fraction step order",
  "tasks": [
    "Review how to find a common denominator",
    "Complete 5 structured fraction problems"
  ],
  "parentPrompt": "Show me how you found the common denominator.",
  "successCheck": "Can explain the full step order without help"
}
```

## 10.4 每周计划必须包含
- `dailyGoals`
- `tasks`
- `parentPrompts`
- `successChecks`
- `pauseList`

## 10.5 pauseList（本周不要做什么）
这是高价值内容，必须有。

例如：
- Do not add harder mixed problems yet
- Do not increase worksheet volume this week
- Do not switch methods again before step order is stable

## 10.6 规则库设计
建立 `plan_rule_templates`（可先代码内配置）：

### 例：procedure_gap
- Day 1–2：slow, explicit step reconstruction
- Day 3–4：same-type short practice
- Day 5：mixed practice
- Day 6：error reflection
- Day 7：mini-check

### 例：reading_issue
- 题干拆解
- 条件标记
- 关键词翻译
- 数量关系表达

## 10.7 数据库存储建议
`reports.planJson` 继续使用，但建议新增：
- `planVersion`
- `planRuleCodes[]`

# 11. 质量层需求（Confidence + Review）

## 11.1 目标
保证系统宁愿谨慎，也不乱判。

## 11.2 三档置信度策略

### 高置信度
条件：
- OCR/extraction 稳定
- finding 有足够 evidence
- taxonomy 聚合稳定

处理：
- 自动发正式版报告

### 中置信度
条件：
- extraction 可用，但 evidence 不够强
- 有部分页质量问题
- diagnosis 分数不够稳

处理：
- 发“谨慎版报告”
- 显示“Some conclusions may need clearer pages”
- 建议补拍或再次上传

### 低置信度
条件：
- OCR 差
- problem items 太少
- evidence 不足
- diagnosis 冲突大

处理：
- 进入 `needs_review`
- 不直接展示完整报告
- 要么管理员审核，要么提示重新上传

## 11.3 质量控制规则
- 每个 top finding 至少 2 条 evidence
- 报告中不能出现完全无 evidence 的强结论
- 7-Day Plan 不得与 diagnosis 脱节
- 报告中不得出现作业答案
- parent 版用语必须可读，不允许全是术语

## 11.4 数据库建议新增字段
### analysis_runs
- `qualityTier`
- `needsReviewReason`
- `retryCount`
- `userVisibleStatus`

### reports
- `overallConfidence`
- `qualityFlag`
- `reviewRequired`
