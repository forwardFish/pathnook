# 0. 执行摘要

## 0.1 本版本要解决什么问题

当前 `familyEducation` 仓库已经存在一套完整的 **Creem 支付链路**：

- `package.json` 里已安装 `creem`、`creem_io`
- `.env.example` 中存在 `CREEM_API_KEY`、`CREEM_WEBHOOK_SECRET`、`CREEM_PRODUCT_*`
- 价格页、Billing 页、server actions、checkout route、webhook route、customer portal 都围绕 Creem 实现
- 现有套餐模型已经成立：`one_time` / `monthly` / `annual`

但你现在的商业目标已经变化：

1. **短期主支付切换到 Freemius**
2. **Creem 不删除，只隐藏，不再走主前台路径**
3. **前台页面、支付入口、账单管理、webhook、权限同步，都要切到 Freemius**
4. **已有产品结构尽量不推倒，减少重写**
5. **要保留未来回切 Creem 或并行启用第二支付商的可能性**

因此，`1.4` 的目标不是“加一个支付按钮”，而是把当前系统从：

> **Creem-first billing product**

升级为：

> **Freemius-primary, Creem-standby billing architecture**

---

## 0.2 本版本最终目标

本版本完成后，系统应达到以下状态：

### 用户侧

- `/pricing` 默认只展示 Freemius 购买路径
- `/dashboard/billing` 默认只展示 Freemius 账单与订阅管理
- 用户购买后，权限由 Freemius 结果驱动
- 用户可进入 Freemius 对接的 Billing Portal
- 站内不再暴露任何 Creem 文案、按钮、结算说明、portal 文案

### 代码侧

- Creem 代码 **不删除**
- Creem 路由 **保留但不再作为默认入口**
- 支付核心逻辑抽象为 **provider 层**
- Freemius 成为默认 `billing provider`
- 现有 `checkoutAction` / `customerPortalAction` 尽量复用，内部切换到 Freemius 实现

### 数据侧

- 继续使用你当前 **Vercel 一体化数据层** 保存 Freemius entitlement / license / subscription / portal 所需映射
- webhook 成为订阅状态同步的权威来源之一
- 权限状态继续以 Pathnook 自己的数据层 entitlement 逻辑为准，而不是在前端直接信任第三方状态

---


## 0.2.1 部署现实边界（本版本强约束）

本版本必须明确采用你当前已经在使用的部署方式：

- **前端与服务端一体部署在 Vercel**
- **没有单独自建后端服务**
- **没有额外再搭一个“后端数据库系统”**
- 现有数据层继续使用项目当前的数据库接入方式（仓库中可见 `drizzle-orm` 与 `@neondatabase/serverless`，结合你的说明，应按“Vercel 一体化数据层 / Vercel 托管数据库接入”理解）
- 所有 Freemius checkout / portal / webhook / entitlement 同步逻辑，必须通过 **Next.js Route Handlers / Server Actions** 落在现有应用内部完成，而不是额外起一个独立 Node 服务

因此，`1.4` 的“数据层改造”含义是：

> **在你现有的 Vercel 数据层里新增或改造 billing 相关表和字段，而不是让你再单独搭一套后端数据库。**

这条约束优先级高于本文其他实现描述。后面凡是提到“本地数据库”“后端同步”“服务端处理”，都统一按 **“当前 Next.js 应用运行在 Vercel 上，使用现有数据库连接与 Route Handlers 完成”** 来理解。

## 0.3 本版本不做什么

本版本明确不做：

- 不重构 FamilyEducation 的学习诊断主业务
- 不改动 AI 诊断、报告、7 天计划、deck、walkthrough 主链路
- 不删除 Creem SDK、Creem route、Creem 配置
- 不同时接入多个前台支付商
- 不一次性做完 Freemius 全量 React Starter Kit UI
- 不做联盟、自动化邮件、复杂营销自动化
- 不做大规模财务后台
- 不把产品重新包装成 Freemius Starter Kit Demo

本版本只做：

> **在现有 FamilyEducation 产品中，把 Freemius 接成主支付与主账单系统，同时把 Creem 暂时隐藏为备用通道。**

# 1. 当前代码审计结论（针对 billing 部分）

## 1.1 仓库与栈现状

当前仓库 `forwardFish/familyEducation` 是一个基于 Next.js App Router 的 TypeScript 项目，仓库 README 明确产品定位为 parent-first 的家庭教育诊断产品，并已完成 landing / pricing / share / PDF / tutor handoff / reminders / admin review 等交付闭环。  
代码层面从 `package.json` 可见：

- `next@15.6.0-canary.59`
- `drizzle-orm` / `drizzle-kit`
- `@neondatabase/serverless`
- `@vercel/blob`
- 已安装 `creem` 与 `creem_io`

这说明当前项目已经是 **Next.js + Drizzle + Vercel Runtime + serverless 数据库驱动 + Vercel Blob + Creem** 的可运行 SaaS 骨架。结合你的补充，本版本应按“**Vercel 一体化应用与数据层**”来设计，而不是按“独立前端 + 独立后端服务 + 独立数据库集群”来设计。

- `app/api/creem/checkout/route.ts`
- `app/api/creem/webhook/route.ts`

### 已存在的支付实现文件

当前仓库已经存在以下支付核心实现：

### 已存在的套餐模型

当前 `catalog.ts` 已定义 3 个套餐：

并且这些套餐已经有：

- `productId`
- `priceId`
- `name`
- `description`
- `unitAmount`
- `interval`
- `ctaLabel`
- `features`

这意味着当前前台价格页和 billing 权限逻辑，已经不是“空壳页”，而是有真实套餐模型可以直接迁移。

## 1.3 当前用户侧文案问题

当前仓库前台仍然是明显的 Creem-first：

### Pricing 页

现有价格页中直接写了：

- `Creem-hosted secure checkout`
- `Checkout is hosted by Creem`
- 计划页与购买说明围绕 Creem 展开

### Billing 页

现有 Billing 页标题和说明明确写的是：

- `Creem billing and unlock status`
- `Payments are hosted by Creem`
- portal 标签也由 `getPortalSupportLabel()` 返回 Creem 文案

这说明当前系统如果直接上线 Freemius，会产生严重的前后台不一致，因此 1.4 必须同步做：

- 页面文案替换
- provider 切换
- portal 切换
- checkout 与 webhook 切换

---

## 1.4 当前数据命名问题

从现有 Creem checkout / webhook 代码可以看到，系统内部把一些第三方字段仍然按 `stripeCustomerId` / `stripeSubscriptionId` 之类的名字传递。  
这虽然在 Creem 时代可以工作，但在 1.4 阶段会出现两个问题：

1. **字段语义已经不准确**
2. **会阻碍未来多 provider 并存**

因此 1.4 必须做一层：

> **provider-neutral billing identity abstraction**

也就是把“第三方支付标识”从 Stripe/Creem 风格，抽象成通用 provider 结构。

# 2. 产品目标与原则

## 2.1 产品目标

1. **Freemius 成为主支付提供方**
2. **现有购买和账单体验尽量不推倒**
3. **Creem 暂时隐藏但保留代码**
4. **用户权限继续由本地 entitlement 控制**
5. **满足 Freemius 审查与长期运营要求**
6. **后续可以低成本重新启用 Creem 或扩第二支付商**

---

## 2.2 核心原则

### 原则 1：前台只能有一个主支付商
当前版本前台只允许看到 Freemius，不允许 Freemius/Creem 并存造成混乱。

### 原则 2：隐藏不等于删除
Creem 保留在代码里，但不允许出现在：

- 首页
- Pricing
- Billing
- Portal CTA
- Footer 支付说明
- 法律页支付说明
- 帮助文档默认说明

### 原则 3：对外路由尽量稳定
用户仍然使用：

- `/pricing`
- `/dashboard/billing`

尽量不改动核心页面 URL，减少前台返工。

### 原则 4：支付 provider 必须可替换
不能再把任何付款逻辑直接写死在 `creem.ts` 风格中。  
必须建立：

- `billing provider interface`
- `freemius provider`
- `creem provider (standby)`

### 原则 5：应用内 entitlement 是最终权限判断层
第三方支付结果不是最终产品权限。  
当前应用在 Vercel 上运行，因此最终控制权仍在 **应用自己的数据层 entitlement**，而不是在前端直接信任 Freemius：

- 报告是否解锁
- 月付/年付是否有效
- credit 是否扣减
- billing portal 是否开放

---

# 3. 最终方案结论

## 3.1 本版本推荐方案

### 推荐方案：Freemius Hosted Checkout + Freemius Customer Portal + Freemius Webhooks + 应用内 Entitlement

这是当前最稳、最适合现有项目的方案。这里的“应用内 entitlement”明确指：**仍然使用当前部署在 Vercel 上的 Next.js 应用和其现有数据层来保存权限与账单映射，不新增独立后端服务。**

### 不推荐方案

#### 不推荐 1：直接把当前前端全部换成 Freemius Starter Kit
原因：
- 当前项目已有自己的 landing / pricing / billing 页面
- 完全替换会引起大量无必要 UI 改造
- 成本高，收益不成比例

#### 不推荐 2：同时保留 Creem 和 Freemius 前台入口
原因：
- 会严重影响信任与转化
- 审查风险高
- 用户会搞不清哪一个才是正式支付入口

#### 不推荐 3：先接 Portal 再接 Checkout
原因：
- 没有稳定 purchase path，Portal 没有意义
- 应先让 Freemius 首单闭环跑通，再补完整 billing self-service

---

## 3.2 分阶段实施策略

### Phase 1（必须完成）
- Freemius checkout
- Freemius redirect processing
- Freemius webhook
- 本地 entitlement 同步
- Pricing 文案改造
- Billing 文案改造
- Creem 前台隐藏
- 法律页 / 联系页 / 支付说明完善

### Phase 2（建议紧随）
- Freemius customer portal
- Billing 页面内嵌或跳转管理
- 计划变更 / 取消 / 发票查看
- upgrade / annual upgrade flow

### Phase 3（后续）
- overlay checkout
- paywall 组件增强
- one-time credits 更细粒度 top-up
- coupon / trial / affiliate
- 营销自动化

# 5. Freemius 产品模型设计

## 5.1 Freemius 后台产品类型

在 Freemius 后台中，本项目必须创建：

> **一个 SaaS Product**

并使用当前已存在的价格结构进行映射。

---

## 5.2 套餐映射方案

当前 FamilyEducation 已有 3 档：

### 1. One-Time Diagnosis
- 本地 `planType = one_time`
- 语义：购买一次诊断或增加一次解锁额度
- Freemius 中映射为：一次性 pricing / top-up / non-recurring purchase
- 本地效果：`reportCredits +1`

### 2. Parent Weekly
- 本地 `planType = monthly`
- 语义：月订阅，订阅期内所有当前报告可解锁
- Freemius 中映射为：monthly subscription plan
- 本地效果：active recurring entitlement

### 3. Parent Annual
- 本地 `planType = annual`
- 语义：年订阅，订阅期内所有当前报告可解锁
- Freemius 中映射为：annual subscription plan
- 本地效果：active recurring entitlement

---

## 5.3 Single Subscription 策略

本版本默认采用：

> **单用户单活跃订阅**

即：

- 同一用户不能同时拥有多个活跃 recurring subscription
- upgrade / downgrade 在同一 license 上完成
- one-time diagnosis 不视为 recurring subscription

这样做的好处是：

- 与 Freemius 默认逻辑一致
- 适合当前家庭教育产品
- 可以降低客服与对账复杂度
- 后续做 yearly upgrade 更清晰

---

## 5.4 Upgrade Flow 要求

Freemius 官方 checkout 支持在已有 `licenseId` 的前提下创建 upgrade checkout，并自动处理 proration。  
因此 1.4 必须要求：

- 若用户已有 monthly entitlement，再购买 annual，不创建新订阅，而是走 upgrade flow
- upgrade flow 必须带现有 `licenseId`
- 本地 entitlement 必须保留 upgrade 前后的映射关系
- UI 上显示为“Upgrade to Annual”，而不是再买一个新套餐

---

# 6. 数据层设计（基于现有 Vercel 数据层，不新增独立后端服务）

## 6.1 目标

本版本不能继续只靠 provider-specific 字段零散拼装 billing 状态。  
必须在**现有 Vercel 数据层**里新增统一的 billing 数据结构。

这里再次强调：

- 不是新建独立后端
- 不是新建另一套数据库系统
- 不是要求你再部署一个 billing microservice

而是：

- 继续沿用当前项目已有的 Drizzle schema / migration 方式
- 继续沿用当前项目在 Vercel 上的 Route Handlers / Server Actions
- 只是在现有 schema 中补上 billing 相关表与字段

---

## 6.2 新增表：`billing_provider_accounts`

用途：记录当前应用用户/团队与第三方 provider 身份映射。

> 这里的“新增表”指的是：在你现在已经接入的数据库里加表，而不是让你另外买一个数据库。

```ts
billing_provider_accounts
- id
- userId
- teamId
- provider                // freemius | creem
- providerUserId          // Freemius userId / Creem customerId 等
- providerCustomerId      // 若 provider 区分 userId/customerId，则保留
- email
- lastSyncedAt
- rawJson
- createdAt
- updatedAt
```

### 约束
- `(provider, providerUserId)` 唯一
- `(provider, providerCustomerId)` 可唯一（如适用）
- 同一 userId 可存在多个 provider account

---

## 6.3 新增表：`billing_entitlements`

用途：应用内权限真相层。

```ts
billing_entitlements
- id
- userId
- teamId
- provider
- localPriceId
- planType                // one_time | monthly | annual
- kind                    // one_time | subscription | topup
- status                  // active | canceled | expired | trialing | unpaid
- providerLicenseId
- providerSubscriptionId
- providerPlanId
- providerPricingId
- providerPurchaseId
- sourceCheckoutId
- currentPeriodStartsAt
- currentPeriodEndsAt
- cancelAt
- canceledAt
- creditsGranted
- creditsRemaining
- lastEventType
- rawJson
- createdAt
- updatedAt
```

### 约束
- `providerLicenseId` 唯一（如存在）
- `sourceCheckoutId` 可索引
- `(userId, provider, localPriceId, status)` 建索引
- recurring 记录要求 `providerLicenseId` 非空

---

## 6.4 新增表：`billing_webhook_events`

用途：记录 webhook 处理日志，支持重放与排错。

```ts
billing_webhook_events
- id
- provider
- providerEventId
- eventType
- payloadJson
- processingStatus        // received | applied | ignored | failed
- errorMessage
- processedAt
- createdAt
```

### 约束
- `(provider, providerEventId)` 唯一

---

## 6.5 当前表兼容策略

如果现有系统里已经有：

- `reportCredits`
- `stripeCustomerId`
- `stripeSubscriptionId`
- billing snapshot
- unlock state

那么 1.4 不要求一次性删掉旧字段。  
但要求：

### Phase 1
- 新 entitlement 表开始写入
- 旧字段仍可读
- billing snapshot 读取优先从新 entitlement 表生成

### Phase 2
- 旧字段逐步变成 derived / compatibility fields
- 前台不再直接依赖旧 provider-specific 字段

# 8. API 设计

## 8.1 必须新增的 Freemius 官方风格端点

按照 Freemius 通用集成模式，本项目必须提供下列端点，但它们都应作为 **当前 Next.js 应用里的 Route Handlers** 存在：

- `/api/checkout`
- `/api/portal`
- `/webhook`

这三条线是 Freemius SaaS 集成的标准核心。

---

## 8.2 `/api/checkout`

### 用途
- 处理 hosted checkout redirect
- 处理 checkout 成功后的 purchase 同步
- 作为 official Freemius checkout processor route

### 部署要求
- 必须部署在当前 Vercel 应用中
- 不额外拆独立 server
- 允许使用 edge / node runtime，但要以 Freemius SDK 与签名校验兼容性为准

### 请求方式
- `GET`
- `POST`（保留，兼容 SDK processor 形式）

### 行为要求
1. 调用 `freemius.checkout.processRedirect(currentUrl, proxyUrl?)`
2. 验证 redirect 合法性
3. 取得 `licenseId` / redirect info
4. 再查询完整 purchase/license 信息
5. 写入本地 entitlement
6. 跳转到 `/dashboard/billing?checkout=success...`
7. 若失败，跳转 `/dashboard/billing?checkout=invalid`

### 注意
- 必须支持 proxyUrl 修正，避免 Vercel / proxy 场景下签名校验失败
- 必须可区分 sandbox 与 production

---

## 8.3 `/api/portal`

### 用途
- 为 Freemius Customer Portal 提供安全 endpoint
- 支持用户在站内查看订阅、发票、付款方式、取消订阅

### 部署要求
- 仍然落在当前 Vercel 应用内
- 使用现有登录态与 session 体系
- 不新建独立 billing backend

### 行为要求
1. 当前用户必须已登录
2. 从本地 provider account / entitlement 中找到 `providerUserId`
3. 用 `freemius.customerPortal.request.createProcessor()` 实现 GET / POST
4. 未找到 Freemius 账户时，返回友好提示，不抛裸错误
5. portal 页面如果未配置，前台必须展示替代说明

---

## 8.4 `/webhook`

### 用途
- 接收 Freemius 平台推送事件
- 同步应用内 entitlement
- 作为订阅状态与 license 状态的权威同步通道之一

### 部署要求
- 继续使用当前应用的 Route Handler 接 webhook
- 继续把事件写入现有数据库
- 不额外搭一个 webhook relay 服务，除非后续规模需要

### 实现要求
1. 使用 `freemius.webhook.createListener()`
2. 注册 handler：
   - `license.created`
   - `license.updated`
   - `license.cancelled`
   - `license.expired`
   - `license.plan.changed`
   - 至少一个订阅失败类事件，例如 `subscription.renewal.failed`
3. 用 `freemius.webhook.createRequestProcessor(listener)` 或 `processFetch`
4. 默认使用签名校验
5. 若平台环境无法拿 raw body，允许切换 API 验证 + secret token 模式
6. 所有 webhook 都先入库 `billing_webhook_events`
7. 幂等处理：同一个 `providerEventId` 不能重复应用

## 8.5 兼容保留端点

### `/api/billing/checkout-session`
保留，但改造成内部 facade：

### `/api/creem/checkout`

- 可返回 404 / 410
- 或仅内部环境可访问

### `/api/creem/webhook`
保留代码，不作为生产主 webhook

# 12. 权限映射规则

## 12.1 One-Time Diagnosis
当 Freemius one-time purchase 成功时：

- 新增或更新一条 `billing_entitlements`
- `kind = one_time`
- `creditsGranted = 1`
- `creditsRemaining += 1`
- 解锁策略由现有 `applyCheckoutCompletionForUser` / billing snapshot 逻辑继续承接

---

## 12.2 Monthly / Annual
当 Freemius recurring subscription 激活时：

- `kind = subscription`
- `status = active` 或 `trialing`
- `currentPeriodEndsAt` 写入
- 本地所有当前报告视为可解锁
- 周复盘 flow 保持开放

---

## 12.3 Cancel / Expire
当收到取消、过期、失败事件时：

- entitlement 状态更新
- portal 页面实时体现
- 已经生成的历史报告不删
- 新增解锁权限停止
- 是否保留当前周期内访问，以 provider 状态和本地周期结束时间共同决定

---

## 12.4 Upgrade
当 monthly -> annual upgrade 时：

- 不新建第二个活跃 recurring entitlement
- 更新原 entitlement 的 `providerPlanId/providerPricingId/currentPeriodEndsAt`
- 本地记录一条 upgrade audit event
- UI 成功页提示必须写 upgrade 结果，而不是“new purchase”

# 18. 文档修订补充（Vercel 一体化部署版说明）

## 18.1 本次修订要点

本次修订专门针对一个容易误解的点做了统一澄清：

- 你当前不是“前端站点 + 独立后端服务 + 单独数据库”模式
- 你是“**Next.js 全栈应用跑在 Vercel 上，数据层也走当前已接入的数据库**”模式

因此本文件里所有新增的：

- entitlement 表
- webhook 事件表
- provider account 表
- checkout / portal / webhook 端点

都应该理解为：

> **在现有 Vercel 应用和现有数据库接入方式里完成，不新增独立后端系统。**

## 18.2 给开发的直接约束

开发时必须遵守：

1. 不要为了 Freemius 再单独搭一个 backend repo。
2. 不要为了 Freemius 再单独引入一套独立数据库。
3. 继续在当前项目里通过 Drizzle schema / migration 管理 billing 表。
4. 继续通过 Next.js Route Handlers 暴露 `/api/checkout`、`/api/portal`、`/webhook`。
5. 继续在当前 Vercel 部署体系里处理 webhook 与 entitlement 同步。
6. Creem 代码保留，但前台入口默认隐藏。

## 18.3 最小实现理解

如果按最小版本落地，真正新增的不是“新后端”，而是：

- 3 个 Route Handlers
- 2~3 张 billing 相关表
- 1 套 provider 抽象
- 价格页 / billing 页文案与按钮切换
- webhook 到 entitlement 的同步逻辑

也就是说，这次更接近：

> **在现有应用里加一层 billing capability**

而不是：

> **重建一套支付后端系统**
