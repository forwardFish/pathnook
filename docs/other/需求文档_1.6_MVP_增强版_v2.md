# A. 为什么要出“增强版”

原版 `1.6_MVP` 已经把 **最小闭环** 讲清楚了：  
先做出能真实收费、真实使用、能完成一次再回来一次的最小闭环。

这个方向没有错。

但如果 MVP 只有：

- Diagnosis
- Evidence
- 7-Day Plan
- Compare（轻版）
- Resume（轻版）

那么用户会觉得它像：

- 更聪明的诊断工具
- 更好的周计划工具
- 更完整的家庭教育报告工具

而未必会立刻感受到：

> **Pathnook 真正想做的是：把高手的学习方式，变成普通人也能调用的 AI 系统。**

所以这份增强版要做的，不是推翻原 MVP，而是在不显著增加工程复杂度的前提下，补上 **最能形成差异化感知的 4 个能力**：

1. **骨架诊断**  
2. **最短能力路径**  
3. **输出关卡**  
4. **高频纠偏**

这四个能力一旦加入，原来的 MVP 就会从：

> 看清问题 → 拿到计划 → 下周回来

升级成：

> 看清骨架 → 找到当前最短能力路径 → 通过输出暴露真实水平 → 通过高频纠偏快速进步 → 下周比较并续接

这就是“最小差异化闭环”。

# B. MVP 的升级目标：从“最小闭环”变成“最小差异化闭环”

## B.1 原 MVP 的价值
原 MVP 解决的是：

- 家长愿不愿意上传真实资料
- 家长是否认可 Diagnosis + 7-Day Plan
- 用户是否愿意付费
- 用户是否会回来第二次

这些仍然成立，不能删。

## B.2 增强版新增的价值
增强版新增的不是更多页面，而是用户第一次就能感知到的不同：

### 价值 1：不是只告诉你“哪里错了”，而是告诉你“当前真正卡在哪个骨架节点”
这就是 **骨架诊断**。

### 价值 2：不是只给一周计划，而是告诉你“为什么先做这一步，这一步是最短能力路径”
这就是 **最短能力路径**。

### 价值 3：不是继续让你看，而是尽快让你“输出一次”，暴露真实水平
这就是 **输出关卡**。

### 价值 4：不是一周后才知道有没有用，而是在当前任务结束后就得到修正
这就是 **高频纠偏**。

---

# C. 增强版新增的 4 个核心能力

## C.1 骨架诊断（Skeleton Diagnosis）

### C.1.1 它解决什么问题
普通用户看到的是：
- 这题错了
- 那题也错了
- 这个知识点不会

高手看到的是：
- 这门内容的骨架是什么
- 当前真正缺的是哪一环
- 哪些问题是主干问题，哪些只是枝叶问题

所以 MVP 第一差异化能力必须是：

> **系统不只是看错题，而是要指出：当前学习内容的骨架，以及当前孩子卡住的核心节点。**

### C.1.2 MVP 最少输出什么
在原有 Diagnosis 基础上，新增 4 个字段：

- `skeletonFocus`：当前最关键的骨架环节
- `whyThisMatters`：为什么这个环节比其他问题更值得先抓
- `ignoreForNow`：当前先不要抓什么
- `leverage20`：最值得先抓的关键 20%

### C.1.3 页面落点
#### 报告页 `/dashboard/reports/[reportId]`
Diagnosis 区在原有字段基础上新增：

- Skeleton focus
- Why this is the leverage point
- Ignore for now

#### 样例页 `/sample-report`
必须新增一块：
- `What the system identifies as the real learning bottleneck`

### C.1.4 实现策略
MVP 阶段不需要单独新表。  
优先做法：

- 在 `diagnosis_outlines.primaryJson`
- `diagnosis_outlines.patternIssuesJson`
- `diagnosis_outlines.thisWeekFocus`

基础上增加扩展字段即可，例如：

- `skeletonFocus`
- `leverageReason`
- `ignoreForNow`

即：**先复用 diagnosis object，不为这一步单独增加复杂 schema。**

---

## C.2 最短能力路径（Shortest Ability Path）

### C.2.1 它解决什么问题
普通计划是：
- 周一做什么
- 周二做什么
- 周三做什么

高手路径是：
- 为什么先做这一步
- 这一小步解决哪个核心能力问题
- 做完这一小步，能力会发生什么变化

所以 Pathnook 必须从“计划器”升级为：

> **最短能力路径生成器。**

### C.2.2 MVP 最少输出什么
在原有 `7-Day Plan` 之前，增加一张“路径卡”：

- 当前最短路径名称
- 为什么先走这一步
- 这一阶段唯一目标
- 达成这一阶段后，下一步是什么

建议命名：

- `Shortest Path`
- `Ability Path`
- 或中文语义对应的 `最短能力路径`

### C.2.3 页面落点
#### 报告页
在 `Diagnosis` 与 `7-Day Plan` 之间新增一个模块：

- `Shortest Ability Path`

#### Child detail 页
在 `Current Focus` 下新增简版路径状态：

- 当前路径
- 当前处于第几步
- 下一步是什么

### C.2.4 实现策略
MVP 阶段不强制新增独立表。  
优先做法：

- 在 `seven_day_plans.focusJson`
- `seven_day_plans.daysJson`

里增加：
- `pathName`
- `pathReason`
- `stageGoal`
- `nextStageHint`

这样可以在不增大工程成本的前提下，把计划从“日程”升级成“能力路径”。

---

## C.3 输出关卡（Output Gate）

### C.3.1 它解决什么问题
普通学习产品默认用户一直输入。  
高手学习方式会很早让用户输出，因为：

- 不输出，就容易误以为自己懂了
- 一输出，真实短板会立刻暴露

所以 Pathnook 的第三个差异化点必须是：

> **不是让用户继续吸收，而是尽早通过一个小关卡，逼出真实水平。**

### C.3.2 MVP 最少输出什么
每周计划里至少有 1～2 天，被系统标成：

- `Output Gate`

每个输出关卡至少包含：
- `gateTitle`
- `whyThisGate`
- `requiredOutput`
- `passCondition`
- `commonFailurePattern`

### C.3.3 输出关卡形式（教育场景）
MVP 阶段只做 4 类，足够：

1. **Explain**：用自己的话解释一道题为什么这样做  
2. **Rebuild**：把错误步骤重建成正确顺序  
3. **Transfer**：做一个变式题  
4. **Spot the mistake**：指出哪一步开始错了

### C.3.4 页面落点
#### 报告页 7-Day Plan 区
把其中 1～2 个 daily card 标为：
- `Output Gate`

#### Child detail 页
Next best action 如果当前是关卡日，要显示：
- `Complete today’s output gate`

### C.3.5 实现策略
MVP 不先做复杂交互引擎。  
先做成：

- 计划卡片中的特殊任务类型
- 后端只生成关卡定义
- 前端先不做复杂判卷，只先提供输出说明与完成判断标准

即：**先做关卡定义，再逐步做关卡执行与评分。**

---

## C.4 高频纠偏（High-Frequency Correction）

### C.4.1 它解决什么问题
普通人学习慢，常常不是因为不学，而是：

- 反馈太晚
- 错误暴露太慢
- 纠偏太慢

所以第四个差异化点必须是：

> **系统不只给长期报告，而是在当前关卡或当前周目标内，尽快指出偏差并修正。**

### C.4.2 MVP 最少输出什么
先不做复杂实时系统。  
MVP 只要求在以下两处体现“高频纠偏”：

#### 场景 A：Diagnosis → 7-Day Plan
系统明确写出：
- 当前最容易重复的错误
- 当前最该避免的错误
- 当前最值得关注的修正动作

#### 场景 B：Compare with last report
系统明确写出：
- 哪个问题减轻了
- 哪个问题没变
- 下次仍需重点盯哪一类错误

### C.4.3 页面落点
#### 报告页
- Diagnosis 区新增 `Most likely repeat pattern`
- Compare 区新增 `What still needs correction`

#### Child detail 页
- Resume 区新增 `Last known correction target`

### C.4.4 实现策略
MVP 阶段依然先不单独建实时纠偏引擎。  
先用：
- diagnosis_outlines
- compare summary
- current focus

三处组合，形成“轻量高频纠偏”的感觉。

## E.3 Output Gate 的最小结构
建议直接放进 `daysJson` 中，先不拆表：

```json
{
  "dayIndex": 3,
  "minutes": 20,
  "goal": "Explain the regrouping order clearly",
  "taskType": "output_gate",
  "gate": {
    "gateTitle": "Explain the full subtraction sequence",
    "whyThisGate": "This exposes whether the child truly understands the process.",
    "requiredOutput": "Explain the full order aloud before solving",
    "passCondition": "Can state the correct step order without prompting",
    "commonFailurePattern": "Jumps into calculation before setting the order"
  }
}
```

# G. 这一版增强之后，MVP 的判断标准

增强版 MVP 不只要求用户觉得“能用”，还要至少满足下面 4 条里的 2 条：

1. 用户第一次看报告时，明确感觉：
   - “系统抓到的不是表面错题，而是真正卡点”

2. 用户看到计划时，明确感觉：
   - “这不是泛计划，而是最短能力路径”

3. 用户执行任务时，明确感觉：
   - “系统在逼我输出真实水平，不是在让我继续看”

4. 用户第二次回来时，明确感觉：
   - “系统记得我上次卡在哪，也知道下次该纠正什么”

### 必须固定为 5 块
1. Diagnosis
2. Evidence
3. 7-Day Plan
4. Compare with last report（如果存在）
5. Share

### Diagnosis 最少内容
- Primary issue
- Secondary issue
- Pattern vs one-off
- What not to overreact to
- This week’s focus

### Evidence 最少内容
- 2–5 条关键 evidence
- page / problem refs
- short why this matters

### 7-Day Plan 最少内容
- day 1–7
- minutes
- goal
- tasks
- parent prompt
- success check
- what not to push this week

### Compare 最少内容
- improved
- unchanged
- needs more support

### Share 最少内容
- share link
- tutor summary preview
- revoke / expiry（轻版即可）

### 不做
- 单独 evidence 强依赖页
- 单独 plan 强依赖页
- walkthrough 主体验

# 9. MVP 验收标准

## 9.1 产品验收
以下 6 条全部成立，MVP 就算成立：

1. 用户能用 Freemius 完成真实付费
2. 用户第一次上传后，能看到清晰判断
3. 用户第一次上传后，能看到 7-day plan
4. 用户第二次上传后，能看到最小 compare
5. 用户回来时，child detail 能告诉他下一步
6. 用户可以把当前结果分享给 tutor

## 9.2 工程验收
1. 不再完全依赖 reportJson blob
2. diagnosis / plan / share 至少有核心对象落表
3. Freemius 付款与解锁正确
4. smoke 能跑通：upload -> report -> pay -> share -> second report compare

# 10. 最后一句

这版不是“把 Stage 1 完整做完”。

它真正要做的是：

> **用最短时间，验证家长是否愿意为“看清问题 → 知道这周做什么 → 下周还能继续看变化”这个闭环持续使用和付费。**

只要这个成立，后面的增强才有意义。
