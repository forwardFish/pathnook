# 0. 执行摘要

## 0.1 这版文档为什么要重写
之前的 `1.6` 更像“Stage 1 完整闭环开发版”，范围偏大。  
你现在要的是：

> **快速验证：家长是否愿意为“判断—行动—反馈—续接”这个闭环持续使用和付费。**

所以这版文档必须把范围主动砍小，只保留真正影响验证的页面、功能、接口和数据对象。

---

## 0.2 这版 MVP 的一句话目标

> **先做出最小闭环，再基于真实付费与复用数据继续改善。**

---

## 0.3 这版 MVP 只验证 4 件事

### 验证 1：家长愿不愿意上传真实学习资料
用户是否愿意：
- 上传最近作业 / quiz / test
- 填最少量上下文
- 等待一份诊断结果

### 验证 2：家长是否认可“判断 + 行动”价值
用户是否觉得：
- Diagnosis 说中了问题
- 7-Day Plan 真能执行
- 不是一份 AI 套话报告

### 验证 3：用户是否愿意付费
必须验证：
- One-time diagnosis 是否能卖出去
- 看到结果后，是否愿意升级 monthly

### 验证 4：用户是否会回来第二次
必须验证：
- 第二次上传是否发生
- 用户是否会看 compare
- 系统是否能接住上一次状态

---

## 0.4 这版 MVP 明确不做什么
以下内容全部后置：

- 复杂 walkthrough / player
- 独立 evidence 深页强依赖
- 独立 plan 深页强依赖
- 完整趋势大盘
- quarterly / annual 成长档案
- tutor workspace
- H5 / PPTX export
- 复杂 admin review 后台
- 学生端
- 成人成长前台
- 多学科扩张
- Creem 前台并存

# 2. MVP 闭环定义

## 2.1 判断（Judgment）
主报告页必须能看到：

- Primary issue
- Secondary issue
- Pattern vs one-off
- What not to overreact to
- This week’s focus

## 2.2 行动（Action）
主报告页必须能看到：

- 7-day plan
- 每天时长
- 每天目标
- 每天 task
- parent prompt
- success check
- what not to push this week

## 2.3 反馈（Feedback）
从第二份 report 开始必须能看到：

- compare with last report
- what improved
- what stayed the same
- what still needs support

## 2.4 续接（Resume）
用户回来时必须能看到：

- current focus
- latest report
- next best action
- re-upload CTA

分享页必须能支持：

- diagnosis summary
- weekly focus
- tutor-ready summary
- revoke / expiry

---

# 3. 支付与商业闭环

## 3.1 Freemius 是唯一主支付
本版 MVP 必须正式采用：

> **Freemius = 唯一公开前台支付路径**  
> **Creem = 隐藏备用，不在任何前台页面出现**

## 3.2 MVP 前台建议只主推 2 档
### 主推
1. One-time diagnosis
2. Monthly family plan

### 可保留但降权
3. Annual family plan

原因：
- 先验证首单
- 再验证“愿不愿意继续回来”
- 不先把重点放在年费

# 6. MVP API（只保留必须接口）

# 7. MVP 开发顺序

## Phase 1：先让第一次结果成立
1. Freemius 主支付切换完成
2. 上传页最小 intake
3. problem item 提取
4. taxonomy
5. diagnosis outline
6. 7-day plan
7. 主报告页（Diagnosis / Evidence / Plan）

### 完成定义
- 用户第一次上传后，能真实付费、能拿到报告、能看懂结果

---

## Phase 2：再让第二次回来成立
8. child detail current focus
9. report compare
10. next best action
11. share token / tutor summary

### 完成定义
- 用户第二次回来时，系统能接住上一次状态

---

## Phase 3：最后补少量稳态能力
12. billing 管理页优化
13. smoke test
14. rollback / feature flag
15. 低质量 run 轻门禁

### 完成定义
- 可以小范围对外测试
- 出问题能回滚
- 核心链路稳定

# 9. MVP 验收标准

## 9.1 产品验收
以下 6 条全部成立，MVP 就算成立：

1. 用户能用 Freemius 完成真实付费
2. 用户第一次上传后，能看到清晰判断
3. 用户第一次上传后，能看到 7-day plan
4. 用户第二次上传后，能看到最小 compare
5. 用户回来时，child detail 能告诉他下一步
6. 用户可以把当前结果分享给 tutor

## 9.2 工程验收
1. 不再完全依赖 reportJson blob
2. diagnosis / plan / share 至少有核心对象落表
3. Freemius 付款与解锁正确
4. smoke 能跑通：upload -> report -> pay -> share -> second report compare

---

# 10. 最后一句

这版不是“把 Stage 1 完整做完”。

它真正要做的是：

> **用最短时间，验证家长是否愿意为“看清问题 → 知道这周做什么 → 下周还能继续看变化”这个闭环持续使用和付费。**

只要这个成立，后面的增强才有意义。
