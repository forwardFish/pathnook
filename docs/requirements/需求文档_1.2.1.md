### 目标 A：在不推翻现有系统的前提下，增强专业诊断能力

### 目标 B：让报告真正像“专业诊断书”
也就是让家长第一次看完报告就能明确：
- 主要问题是什么
- 为什么是这个问题
- 哪些是重复模式
- 这周先做什么
- 这周不要做什么

- tutor 独立账号体系

- 企业级 notification center
- 复杂课程图谱与 curriculum mapping
- 自动推荐外部练习题库
- 超复杂趋势引擎和长期 AI 教练层

## 2.3 本版本的正确完成定义
本版本完成，不代表整个产品“全部完成”，而是代表：

> **FamilyEducation 已经从“会生成报告”升级成“有专业诊断能力的系统”。**

# 3.1 5.1 Diagnostic Intake：诊断输入层

- `sourceType`
- `notes`
- upload 文件

要想像“医生问诊”一样做诊断，必须先拿到上下文。

## 本模块要解决什么问题
要解决：

- 这次诊断到底是“考后分析”还是“周复盘”
- 家长最担心的到底是概念、步骤、粗心还是速度

# 3.2 5.2 Problem Item：事实层抽取

## 为什么这是最关键的基础层
如果没有稳定的 `problem_items`，后面的 taxonomy、diagnosis、plan 都会漂。  
所以这一层必须看作：

> **专业诊断的“病历事实层”**

## 本模块要解决什么问题
把上传的 5–10 页材料，转换成一组结构化问题项，而不是让 LLM 直接自由总结。

## 每个 problem item 必须具备的字段
最小结构：

```json
{
  "pageNumber": 2,
  "problemNumber": "Q4",
  "questionSummary": "Fraction addition with unlike denominators",
  "studentResponseSummary": "Added numerators directly without finding a common denominator",
  "teacherMark": "wrong",
  "confidence": 0.82
}
```

## 推荐增强字段
- `sourceSnippet`
- `evidenceAnchor`
- `workType`
- `topicHint`
- `normalizedTeacherSignal`

## 字段定义
### `questionSummary`
不是原题全文复制，而是简短摘要，用于：
- 后续分类
- report 引用
- tutor summary

### `studentResponseSummary`
不是直接写“做错了”，而是要写：
- 学生具体做了什么
- 错在什么动作

### `teacherMark`
枚举：
- `correct`
- `wrong`
- `partial`
- `unknown`

### `workType`
枚举：
- `multiple_choice`
- `computation`
- `open_response`
- `word_problem`
- `geometry`
- `mixed`

### `topicHint`
枚举/自由文本都可，MVP 推荐枚举优先：
- `fractions`
- `equations`
- `geometry`
- `arithmetic`
- `word_problems`
- `unknown`

## 数据库建议
现有 `problem_items` 新增字段：
- `questionSummary`
- `studentResponseSummary`
- `workType`
- `topicHint`
- `sourceSnippet`
- `evidenceAnchorJson`
- `normalizedTeacherSignal`

## API / pipeline
- `POST /api/runs/[runId]/extract-items`
- `GET /api/runs/[runId]/items`

## 与页面的关系
### 运行状态页
显示：
- extracting items
- items extracted
- low extraction confidence warning

### 报告页
Evidence 区块里的所有 finding 都必须能回到 problem item

## 验收标准
- 每个 report 至少抽出可用的 `problem_items`
- `Top Findings` 都能追溯到至少一个 item
- item 可追溯到 page / anchor
- OCR 差时不会伪造 items

# 3.3 5.3 Taxonomy：错误分类层

## 为什么这层是核心壁垒
因为：
- 没有 taxonomy，就没有稳定 diagnosis
- 没有 taxonomy，就没有 weekly compare
- 没有 taxonomy，就没有长期趋势

## MVP taxonomy（固定 8 类）
1. `concept_gap`
2. `procedure_gap`
3. `calculation_slip`
4. `reading_issue`
5. `notation_error`
6. `strategy_error`
7. `careless_slip`
8. `incomplete_reasoning`

## 每个 item error 的输出结构
```json
{
  "errorType": "procedure_gap",
  "severity": "high",
  "confidence": 0.76,
  "rationale": "Student skipped the common denominator step before adding fractions."
}
```

## severity 规则
- `low`
- `medium`
- `high`

### 评分建议
- low：单题轻度偏差，对后续影响小
- medium：出现明显错误，但未必构成模式
- high：系统性错误，影响后续学习

## 多标签策略
MVP 建议：
- 每个 item 最多 2 个标签
- 其中 1 个 `primaryErrorType`
- 可选 1 个 `secondaryErrorType`

避免一题打 4–5 个标签，导致诊断漂移。

## 数据库建议
沿用 `item_errors`，但确保字段齐全：
- `problemItemId`
- `errorType`
- `severity`
- `confidence`
- `rationale`
- `isPrimary`

## 必须加入的约束
- 每个高优先级 finding 必须至少有 taxonomy 支撑
- 无 rationale 的 label 不可进入 report
- confidence 太低的 label 不可直接参与 primary diagnosis 聚合

## 验收标准
- report 中的 Top Findings 都有 taxonomy 支撑
- rationale 能对应 evidence
- 相同类型错误在多页出现时，可稳定聚合成 pattern issue

# 3.4 5.4 Diagnosis 聚合：从错误到判断

## 为什么这是“专业感”的来源
普通工具只会说：
- 这题错了
- 这题应该这样做

真正专业的系统必须回答：
- 主要问题是什么
- 为什么这么判断
- 哪些是模式性问题
- 哪些只是偶发
- 这周不该过度反应什么

## 本模块必须输出
- `primaryDiagnosis`
- `secondaryDiagnosis`
- `patternIssues`
- `sporadicIssues`
- `doNotOverreactPoints`

## Diagnosis 不是简单汇总 taxonomy
必须引入 **issue tree + hypothesis scoring**

## Diagnosis scoring 规则
建议使用：

```text
Diagnosis Score =
(error frequency * 0.4)
+ (severity weight * 0.3)
+ (cross-page repetition * 0.2)
+ (teacher signal support * 0.1)
```

## Pattern issue 判定
满足任一：
- 同 taxonomy 在 2 页以上出现
- 同 taxonomy 出现 >= 3 次
- 与上一次报告同类型问题重复出现

## Sporadic issue 判定
满足：
- 只出现 1 次
- 不构成重复模式
- 对本周主目标不构成关键影响

## Do-not-overreact points
这是报告高级感的重要来源。

系统要主动告诉家长：
- 哪些问题只是偶发
- 哪些暂时不应成为本周主重点
- 不要因为 1–2 个 arithmetic slip 就误判成“基础全盘不行”

## 数据库存储建议
在 `reports` 新增/规范字段：
- `primaryDiagnosisCode`
- `secondaryDiagnosisCodesJson`
- `patternSummaryJson`
- `sporadicSummaryJson`
- `doNotOverreactJson`
- `overallConfidence`

## 页面改造
页面：`/dashboard/reports/[reportId]`

新增区块：
- `Primary Diagnosis`
- `Secondary Diagnosis`
- `Pattern Issues`
- `Sporadic Issues`
- `Do not overreact to`

## 必须达到的用户体验
家长看完后能复述：
- 主要问题是什么
- 为什么是这个问题
- 哪些先别慌
- 本周先抓什么

## 验收标准
- 报告不再只是罗列错误
- 报告首屏更像“判断书”
- report 中的判断和 evidence 一一对应
- 可以明确区分 pattern / sporadic

# 3.5 5.5 7-Day Plan：处方层

## 为什么这层决定用户值不值
用户不是为“分析”付钱，  
而是为：

> **分析之后，知道这一周到底怎么做**

## 原则
不能只靠自由生成。  
必须是：

> **规则模板库 + LLM 补充表达**

## 每天输出结构
```json
{
  "day": 1,
  "minutes": 20,
  "goal": "Rebuild fraction step order",
  "tasks": [
    "Review how to find a common denominator",
    "Complete 5 structured fraction problems"
  ],
  "parentPrompt": "Show me how you found the common denominator.",
  "successCheck": "Can explain the full step order without help"
}
```

## 必须包含的字段
- `day`
- `minutes`
- `goal`
- `tasks`
- `parentPrompt`
- `successCheck`

另外整周必须有：
- `pauseList`

## pauseList 为什么必须有
这是你和普通学习建议工具拉开差距的地方。  
系统必须明确告诉家长：

- 这周不要加大题量
- 这周不要上更难的混合题
- 这周不要频繁换方法
- 这周不要把偶发粗心当主问题

## 规则库设计建议
建议建立 `plan_rule_templates`

### procedure_gap
- Day 1–2：慢速步骤重建
- Day 3–4：同类短练
- Day 5：轻微迁移
- Day 6：错题回看
- Day 7：mini-check

### concept_gap
- Day 1：概念重讲 / 用自己的话解释
- Day 2–3：概念题 + 基础题
- Day 4–5：概念到步骤的连接
- Day 6：混合短练
- Day 7：回顾与小测

### reading_issue
- Day 1：题干拆解
- Day 2：条件标记
- Day 3：数量关系翻译
- Day 4–5：两道同类 word problem
- Day 6：错题复盘
- Day 7：mini-check

## 数据库建议
`reports.planJson` 继续保留，但新增：
- `planVersion`
- `planRuleCodesJson`

## 页面改造
在 `7-Day Plan` tab 中明确按 day card 展示：
- Day number
- minutes
- goal
- tasks
- parent prompt
- success check

## 验收标准
- plan 必须和 diagnosis 对应
- plan 必须具体，不能泛泛
- 家长看到后本周能执行
- 同一个 diagnosis 不同 report 的 plan 不应完全复制粘贴

# 3.6 5.6 Confidence Gate：质量门禁与 Review Gate

## 为什么一定要做
如果不做 confidence gate，你的系统会出现两个大问题：

1. 低质量输入也输出很肯定的结论
2. 报告看起来像真的，但实际是乱判

这是最伤信任的地方。

## 三档质量层级
### A 档：高置信度
条件：
- extraction 稳定
- evidence 足够
- taxonomy 聚合稳定
- diagnosis clear

处理：
- 自动发布正式版报告

### B 档：中置信度
条件：
- extraction 可用，但不够稳
- 部分页模糊或信息不完整
- diagnosis 有倾向，但不够强

处理：
- 发布谨慎版报告
- 增加提示：
  - some conclusions may need clearer pages
  - consider uploading correction pages

### C 档：低置信度
条件：
- OCR 很差
- problem items 太少
- evidence 不足
- diagnosis 冲突大

处理：
- 进入 `needs_review`
- 不直接展示完整报告
- 提示补拍 / 补充订正页 / 等待审核

## 诊断质量门禁（Diagnosis Quality Gate）
新增统一评分表：

### 一级门禁（必须全过）
1. 是否有 `primaryDiagnosis`
2. 是否有 `pattern vs sporadic` 区分
3. 每个 Top Finding 是否至少有 2 条 evidence
4. 是否有 `this week focus`
5. 是否有 `pauseList`
6. 7-Day Plan 是否每天可执行

### 二级门禁（建议项）
1. summary 是否因 intake 场景发生差异
2. tutor share 是否比 parent 版更专业
3. 报告措辞是否避免重复模板
4. 是否明确说明“不必过度反应”的点

## 质量等级建议
- `A`：可直接发布
- `B`：可发布，但建议抽检
- `C`：needs_review
- `D`：不允许发

## 数据库建议
### analysis_runs
新增：
- `qualityTier`
- `needsReviewReason`
- `retryCount`
- `userVisibleStatus`

### reports
新增：
- `qualityGateScore`
- `qualityFlag`
- `reviewRequired`

## 页面改造
### 运行状态页 `/dashboard/runs/[runId]`
增加阶段可视化：
- extracting items
- classifying errors
- building diagnosis
- generating plan
- quality check

### 报告页
如果是 B 档，显示：
- `Cautious report`
- `Some conclusions may become stronger with clearer pages`

如果是 C 档：
- 不进正式 report
- 引导去 `needs_review` / 补拍

## 验收标准
- 不允许无 evidence 强结论
- 不允许低质量输入伪装成高质量输出
- 用户能理解为什么某次 run 被要求补图或复审
