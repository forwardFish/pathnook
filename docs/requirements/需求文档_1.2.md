## 1. 对比结论：旧版 1.2 还差了什么

和这次 Vercel 总结文档对比后，旧版 `需求文档_1.2.md` 主要缺少以下内容：

### 1.1 缺少真实状态盘点
旧版更偏“理想中的专业诊断系统设计”，但没有说明当前仓库里哪些模块已经实现、哪些只是设计、哪些仍然阻塞上线。

### 1.2 缺少 Vercel 生产化要求
旧版没有把以下内容写成明确需求：

- Git 自动部署
- Vercel 项目绑定
- Neon 作为唯一生产数据库
- Blob 作为唯一生产文件链路
- 生产环境变量矩阵
- preview / production 双环境验收

### 1.3 缺少未完成功能清单
旧版没有把“功能未完成”和“功能已完成但未生产验收通过”拆开写，导致后续推进时容易误判。

### 1.4 缺少上线 blocker 列表
旧版没有明确写出当前真正阻塞上线的 4 类问题：

- Google OAuth 未配置
- OpenAI 真实处理未配置
- Creem 真实支付未配置
- Blob 私有文件链路未闭环

### 1.5 缺少最终验收口径
旧版强调产品能力，但没有把“什么叫完成、什么叫可上线、什么叫仍未完成”写成门禁标准。

## 2. 产品目标

FamilyEducation 的目标保持不变：

> 面向欧美家长，提供一个 `parent-first` 的数学学习诊断 Web 产品。  
> 家长上传孩子最近的作业、测验、订正页，系统输出清晰、可信、可执行、可分享的诊断报告与 7-Day Plan。

当前阶段的目标不是再做一个 demo，而是把现有系统推进到：

- 可在 Vercel 上直接部署
- 可直接用生产域名访问
- 不依赖本地 runtime 文件夹、本地数据库、本地 Docker
- 在真实第三方配置补齐后，可跑完整线上链路

## 5. 诊断产品增强需求

这一部分继承旧版 1.2 的核心价值，但改写成“未来要做什么、当前做到什么”的方式。

### 5.1 Diagnostic Intake Card

#### 当前状态
- 上传页已有基础表单
- 已有 `sourceType` 和 notes 等输入

#### 未完成点
- 尚未形成完整的“诊断上下文输入层”

#### 需求
在 `/dashboard/children/[childId]/upload` 新增 `Diagnostic Intake Card`，至少包括：

- `diagnosticGoal`
  - `after_quiz_test`
  - `homework_keeps_failing`
  - `before_hiring_tutor`
  - `weekly_review`
  - `other`
- `recentTrend`
  - `improving`
  - `stable`
  - `declining`
  - `not_sure`
- `teacherFeedbackPresent`
- `parentConcern`
  - `concepts`
  - `steps`
  - `carelessness`
  - `reading_word_problems`
  - `speed`
  - `confidence`
  - `not_sure`
- `hasTutor`
  - `none`
  - `occasional`
  - `regular`
- `notes`

#### 验收标准
- intake 数据可随 upload 一起保存
- run 和 report 层可读取这些字段
- 这些字段会影响 summary / plan 语气和内容

### 5.2 Problem Item 结构增强

#### 当前状态
- `problem_items`、`error_labels`、`item_errors` 等表已存在
- 基础提取链路已存在

#### 未完成点
- 结构化 item 信息还不够强
- 还没有完整达到“专业诊断事实层”的目标

#### 需求
每个 `problem_item` 至少支持：

- `pageNumber`
- `problemNumber`
- `questionSummary`
- `studentResponseSummary`
- `teacherMark`
- `confidence`
- `sourceSnippet`
- `evidenceAnchor`
- `workType`
- `topicHint`

#### 验收标准
- report 中的 finding 能回溯到 item
- item 能回溯到 page / anchor
- item 信息可用于 diagnosis scoring

### 5.3 Taxonomy 与错误分类

#### 当前状态
- 错误分类基础结构存在

#### 未完成点
- taxonomy 深度和可解释性仍不足

#### 需求
MVP taxonomy 至少支持以下大类：

- `concept_gap`
- `procedure_gap`
- `calculation_slip`
- `reading_issue`
- `notation_error`
- `strategy_error`
- `careless_slip`
- `incomplete_reasoning`

每个 item error 至少包含：

- `errorType`
- `severity`
- `confidence`
- `rationale`

#### 验收标准
- 每个 Top Finding 至少有可追溯 taxonomy 支撑
- rationale 不是空话，且能对应 evidence

### 5.4 Diagnosis 聚合层

#### 当前状态
- Diagnosis 页签已存在

#### 未完成点
- 还没有完全把“表面错误”与“根因诊断”分开
- pattern / sporadic / do-not-overreact 尚未形成稳定规则

#### 需求
report 至少输出：

- `primaryDiagnosis`
- `secondaryDiagnosis`
- `patternIssues`
- `sporadicIssues`
- `doNotOverreactPoints`

建议 diagnosis scoring 综合以下维度：

- error frequency
- severity weight
- cross-page repetition
- teacher signal support

#### 验收标准
- 报告不能只罗列错误
- 报告必须能说明为什么得出这个判断
- 报告必须能区分重复问题与偶发问题

### 5.5 7-Day Plan 规则化

#### 当前状态
- 7-Day Plan 页面已存在

#### 未完成点
- 还没有明确的规则模板库
- 计划质量仍可能过度依赖自由生成

#### 需求
7-Day Plan 至少包含：

- `day`
- `minutes`
- `goal`
- `tasks`
- `parentPrompt`
- `successCheck`
- `pauseList`

并建立与 diagnosis code 对应的 plan rule templates。

#### 验收标准
- plan 不能脱离 diagnosis
- plan 必须具体到家长本周能执行
- 计划不能只有泛泛建议

### 5.6 Confidence Gate 与 Review Gate

#### 当前状态
- `needs_review` 路径已存在
- admin review 基础页已存在

#### 未完成点
- 置信度分层与用户可见解释还可以继续强化

#### 需求
建立至少三档质量层级：

- 高置信度：自动发布正式报告
- 中置信度：发布谨慎版报告或提示需更清晰材料
- 低置信度：进入 `needs_review`

#### 验收标准
- 不允许无证据强结论
- 不允许低质量输入伪装成高质量输出
- 用户能理解为什么某次 run 被要求补图或复审
