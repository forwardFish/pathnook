# 0. 执行摘要

## 0.1 这份文档解决什么问题

你现在已经有三条连续思路：

1. **第一阶段**：AI 家庭教育学习成长系统  
2. **第二阶段**：成长引擎  
3. **第三阶段**：海外泛成长产品

问题不在于路线不清，而在于：

- 第一阶段如果做得太窄，后面会推倒重来
- 第一阶段如果做得太泛，家长看不懂、转化会变差
- 1.3 / 1.3.2 / 1.3.3 已经给出了结构化方向，但还缺一份**“开发边界文档”**，告诉开发到底哪些必须现在做、哪些只能做轻版、哪些必须后置

因此，`1.3.4` 的核心任务只有一个：

> **在不破坏“第一阶段必须 parent-first、diagnosis-first”的前提下，把第二阶段未来一定会复用的底层能力抽出来，规定为第一阶段的强制实现项。**

这份文档不是讨论宏大愿景，而是回答下面这些非常工程化的问题：

- 哪些能力属于**第一阶段必须现在做**
- 哪些能力属于**现在做轻版**
- 哪些能力属于**坚决后置**
- 哪些对象必须从一开始就按“成长引擎”建模，而不能写死成“教育专用 JSON”
- 页面、API、数据表、生成链路、质量门禁、分享与回放，应该如何按照这个边界落地

---

### 对产品的要求
**前台必须专注家庭教育，底层必须兼容成长引擎。**

### 对开发的要求
**Stage 1 的页面心智只卖给家长；Stage 2 的底层对象从今天就开始埋。**

---

## 0.3 本文档的最终交付目标

本版本不是交付新页面原型，而是交付：

1. **第一阶段开发边界表**
2. **必须现在做 / 现在做轻版 / 坚决后置** 的完整分类
3. **第一阶段必须采用的通用底层对象**
4. **与 1.3 / 1.3.2 / 1.3.3 一致的 API、DB、生成链路与验收标准**
5. **直接可给 Codex / 开发同学执行的开发约束**

---

## 0.4 本文档最高原则

从 `1.3.4` 开始，所有开发都必须同时满足以下 8 条原则：

### 原则 1：Stage 1 的前台必须 parent-first
首页、样例页、报告页、计划页、分享页的默认心智，必须服务家长，不得滑向学生自学工具或 AI 课堂产品。

### 原则 2：Stage 1 的核心价值仍然是 diagnosis / evidence / 7-day plan
Guided Walkthrough 只是增强层，不得取代静态报告主体验。

### 原则 3：Stage 2 会复用的能力必须现在就按通用层建模
尤其是：
- ProblemItem
- EvidenceAnchor
- DiagnosisOutline
- ExplanationCard
- SevenDayPlan
- Action / Playback
- QualityGate
- ShareArtifact
- Timeline

### 原则 4：任何“好看但不复用”的重功能一律后置
例如：
- PPTX
- 强白板
- 实时多 agent 可视化
- 聊天入口化
- 复杂课堂沉浸模式

### 原则 5：先结构化，再生成内容
继续坚持 1.3 已经确立的两段式生成，不允许回退成“一次性生成整份报告”的黑箱模式。

### 原则 6：任何生成结果必须可校验、可追溯、可降级
没有证据、没有引用、没有质量评分、没有 review gate 的内容，不允许直接面向家长发布。

### 原则 7：共享能力先做轻版，但抽象不能缺
比如 Walkthrough、Weekly Review、Timeline、Share，这些第一阶段可以轻做，但底层对象与接口不能没有。

### 原则 8：第一阶段不能为了“以后扩展”而牺牲现在转化
也就是说：
- 前台文案继续卖 clear diagnosis
- 页面优先级继续卖 evidence-backed findings
- 家长执行帮助继续卖 7-day action plan
- 所有未来扩展都只能藏在底层，不应喧宾夺主

# 2. 第一阶段的正式产品身份

## 2.1 正式产品身份

从 `1.3.4` 开始，第一阶段的产品身份正式定为：

> **面向欧美家长的、基于真实作业证据的每周学习诊断与行动系统。**

它不是：

- AI classroom
- 学生学习 App
- 题库平台
- AI 陪练课
- 知识管理工具
- 通用成长平台

它首先是：

- 家长上传最近 5–10 页孩子数学作业
- 系统给出清晰诊断
- 用证据证明判断
- 给出 7 天计划
- 让家长知道本周盯什么
- 支持 tutor handoff
- 可选进入 walkthrough 理解重点

---

## 2.2 第一阶段的主要用户与次要用户

### 主用户
- 家长
- 为孩子付费
- 需要判断与决策支持
- 需要一周行动建议
- 需要与 tutor 高效沟通

### 次用户
- tutor / tutoring center
- 用分享页快速看懂问题
- 不需要完整后台
- 只需要只读理解、下一次 session 重点、关键证据

### 明确不是主用户
- 学生本人长期操作系统
- 沉浸式 AI 课堂用户
- 泛成长成人用户
- 内容创作者 / knowledge workers

这些人群属于第二阶段或第三阶段，不在第一阶段前台范围内。

# 3. 第一阶段开发边界表（总表）

## 3.1 三类边界定义

### A. 必须现在做
不做就会直接影响：
- 首单转化
- 结果可信度
- 后续第二阶段复用
- 系统稳定性与可持续演化

### B. 现在做轻版
必须有，但只能轻做。  
因为它们有第二阶段价值，但第一阶段不应喧宾夺主，也不应过度消耗开发资源。

### C. 坚决后置
第一阶段不做，或仅留接口，不做完整前台。  
原因通常是：
- 不直接影响转化
- 过度课堂化
- 过度复杂
- 现在做容易冲淡 parent-first 心智

# 4. 必须现在做：详细需求

---

## 4.1 Intake / Upload（必须现在做）

### 目标
保证第一阶段前台能顺畅进入诊断主链路。

### 必须覆盖
- 上传 PDF / 图片
- child 基本信息
- grade / locale
- parent concern
- 本次诊断目的（quiz/homework/weekly review 等）
- materials 数量与质量检测

### 为什么属于“必须现在做”
因为没有 intake，就没有诊断起点。  
而且第二阶段虽然前台会变，但“structured intake”仍然会保留，只是字段换成成长场景。

### 必做字段
- `childId`
- `uploadId`
- `locale`
- `diagnosticGoal`
- `parentConcern[]`
- `notes`
- `materialsSummary`
- `pageCount`
- `uploadQualityStatus`

### 非目标
- 不做复杂问卷
- 不做长表单
- 不要求家长填写大量教育背景

### 验收
- 家长能在短路径内完成上传
- 系统能拿到足够的场景条件用于 DiagnosisOutline
- 低质量上传可被标记，不直接进入自动发布

---

## 4.2 ProblemItem / EvidenceAnchor 事实层（必须现在做）

### 目标
把原始作业材料变成结构化事实层，而不是直接喂给模型写报告。

### 这是第一阶段最核心的强制项
因为第一阶段是教育产品，但第二阶段成长引擎也必须建立在同类“事实单元”上。  
如果这里写死成教育专用长文，后面全部推倒。

### 必须实现的对象

#### `ProblemItem`
表示一个可诊断的问题单元。

最低字段：
- `id`
- `runId`
- `uploadId`
- `anchors[]`
- `questionSummary`
- `studentResponseSummary`
- `teacherMark`
- `workType`
- `topicHint`
- `confidence`

#### `EvidenceAnchor`
表示证据锚点。

最低字段：
- `pageId`
- `pageNumber`
- `bbox`
- `problemNumber`
- `snippet`
- `confidence`

### 必须满足
- 每个 `ProblemItem` 至少 1 个 anchor
- 每个 anchor 必须能回到原始 page
- 前端 evidence 页必须能利用这些锚点跳转/高亮
- 后续 walkthrough/action 也必须复用这些锚点

### 为什么第二阶段也会复用
第二阶段的“成长单元”不是数学题，但也会需要：
- 输入单元
- 支撑证据
- 可回到原始材料
- 可用于 explain / review / share

所以 `ProblemItem + EvidenceAnchor` 不是教育特例，而是通用成长引擎的事实层雏形。

### 验收
- DB 中真实落表
- 可以用于 report 证据页
- 可以用于 explanation card
- 可以用于 walkthrough highlight
- 可以用于 tutor share 跳页

---

## 4.3 Taxonomy / DiagnosisOutline（必须现在做）

### 目标
先判断结构，再生成内容。

### 必须实现的对象

#### `ItemErrorLabel`
- errorType
- severity
- confidence
- rationale
- isPrimary

#### `DiagnosisOutline`
- primary
- secondary
- patternIssues
- sporadicIssues
- doNotOverreact
- overallConfidence
- qualityGrade
- version
- locale

### 为什么属于必须现在做
这是第一阶段“专业性”的核心。  
没有 DiagnosisOutline，系统就还是一个“会写长报告的 LLM”。

### 强制规则
- primary 必须存在
- primary / secondary / pattern / sporadic 必须都能引用 evidenceItemIds
- `doNotOverreact` 必须作为正式前台区块，不再只是后端字段
- qualityGrade 必须影响是否自动发布

### 与第二阶段的关系
第二阶段不会再叫 DiagnosisOutline，但一定会有同类对象：
- 当前核心问题
- 次问题
- 模式性问题
- 偶发问题
- 不该过度反应的点
- 这周先抓什么

所以现在必须按结构化对象建模，而不是隐藏在一篇报告文本里。

### 验收
- 能稳定生成 structured diagnosis
- 引用存在
- 可驱动 report / plan / walkthrough
- 可进入 quality gate

---

## 4.4 Parent Report Sections（必须现在做）

### 目标
让第一阶段产品真正能卖。

### 必须实现的前台主报告结构
报告主页优先级固定为：

1. Diagnosis
2. Evidence
3. 7-Day Plan
4. Weekly Review / Compare（如有）
5. Share
6. Walkthrough（增强）

### Diagnosis 区块必须包含
- Primary issue
- Secondary issue
- Pattern vs sporadic
- What not to overreact to
- This week’s focus

### Evidence 区块必须包含
- 证据分组
- 页码 / 题号
- 简短 why this supports the finding
- tutor 可快速看懂的结构

### 7-Day Plan 必须包含
- 每日目标
- 时长
- tasks
- parent prompt
- success check
- pauseList
- Questions to ask your child this week
- What not to push this week

### Share 区块必须包含
- tutor-ready summary
- share token 入口
- privacy boundary notice

### 为什么属于必须现在做
因为第一阶段不是卖播放器，而是卖：
- diagnosis clarity
- evidence-backed findings
- actionable 7-day plan
- tutor handoff

### 验收
- 不依赖 walkthrough 也能成立
- 不看任何播放功能，家长也能看懂核心价值
- 可直接成为 sample report 的展示基础

---

## 4.5 ExplanationCard（必须现在做）

### 目标
把“讲解”从报告长文中抽出来，变成可复用对象。

### 最低字段
- `id`
- `runId`
- `targetErrorType`
- `anchors`
- `objective`
- `plainExplanation`
- `steps[]`
- `commonPitfall`
- `quickCheck`
- `qualityScore`
- `actions?`

### 第一阶段必须做到的使用方式
- report 内嵌关键讲解卡片
- evidence 页可进入单卡讲解
- walkthrough 轻版可按卡片驱动
- tutor share 可展示关键卡片

### 为什么必须现在做
因为第二阶段成长引擎最可能直接复用的对象，就是这种“带证据的讲解单元”。  
如果现在把讲解写死成 report paragraph，以后最难拆。

### 强制规则
- 卡片必须引用 anchors
- quickCheck 不能为空
- steps 必须可执行，不得空泛
- actions 可以为空，但卡片内容不能为空

### 验收
- 至少能稳定生成 top-3 卡片
- 可独立展示
- 可被 report / walkthrough / H5 share 复用

---

## 4.6 SevenDayPlan（必须现在做）

### 目标
把 diagnosis 转成家长真正能执行的处方。

### 必须字段
- `focus`
- `days[]`
- `minutes`
- `goal`
- `tasks[]`
- `parentPrompt`
- `successCheck`
- `pauseList`
- `questionsToAskChild[]`
- `whatNotToPush`
- `version`

### 为什么必须现在做
因为第一阶段能不能卖，不在于报告有多长，而在于家长看完后：
- 这周该怎么做
- 每天看什么
- 怎么判断有没有变化

### 为什么第二阶段也会复用
第二阶段不一定还是 7 天，但一定还会有“短周期下一步计划”。  
所以 `PlanDay` 这种对象必须先建出来。

### 验收
- 7 天齐全
- 每天可执行
- 家长提示语可直接使用
- success check 清晰
- pauseList 清晰

---

## 4.7 Quality Gate / Admin Review（必须现在做）

### 目标
让系统不是“能生成”，而是“敢发布”。

### 必须有的门禁层
- Schema 校验
- 引用一致性校验
- confidence / qualityGrade
- 发布门禁
- admin review queue

### 最低等级策略
- `A/B`：可直接发布
- `C`：静态谨慎版，不开放增强层
- `D`：进入 `needs_review`

### 为什么必须现在做
教育场景是高信任产品。  
如果没有 Quality Gate，你第一阶段越推广，风险越大。

### 为什么第二阶段也必须用
成长引擎一旦扩场景，输入复杂度只会更高。  
Quality Gate 是基础设施，不是教育专用 feature。

### 验收
- 所有核心对象有 schema 校验
- 引用不存在时报错或降级
- 低质量结果不自动对外开放

---

## 4.8 Tutor-ready Summary / Share（必须现在做）

### 目标
让家长能把判断结果快速交给 tutor。

### 必须有的能力
- tutor-focused summary
- share link
- 可选 tutor walkthrough 入口
- focus areas for next session
- questions to clarify with the child

### 为什么必须现在做
因为 tutor handoff 是第一阶段核心价值之一，不是次要功能。

### 为什么第二阶段也会复用
第二阶段成长引擎会天然需要“把某个成长单元发给他人看”的能力。  
今天发给 tutor，未来可能发给 coach / mentor / collaborator。

### 验收
- share token 可生成
- 只读权限生效
- parent-only 内容不外泄
- tutor 能在 2 分钟内抓住核心重点

---

## 4.9 Privacy / Share Boundary（必须现在做）

### 目标
增强欧美家长信任，避免未成年人数据边界模糊。

### 必须前置显示
- who can access
- expires on
- revoke access
- what data is shown
- view-only notice

### 为什么必须现在做
这直接影响首单转化与信任。  
不是“法务以后再补”的事情。

### 验收
- report / share / walkthrough 都有访问边界说明
- token 可撤销
- 前台有清晰边界文案

## 4.10 Provider / Storage / Jobs 抽象（必须现在做）

### 目标
避免第一阶段后端写死，导致第二阶段无法复用。

### 必须抽象的 provider
- StorageProvider
- ParseProvider
- LLMProvider

### 必须抽象的运行层
- `analysis_runs`
- stage / retry / status
- queued / running / needs_review / done / failed

### 为什么必须现在做
第一阶段就已经涉及：
- upload storage
- parse / OCR
- LLM generation
- export
- async jobs

如果现在写死在 route 里，未来很难扩。

### 验收
- provider 不直接散落在页面/API 中
- run 生命周期可追踪
- 失败可重试
- 成本 / 失败率可监控

# 14. 最后一句

`1.3.4` 的本质不是再加功能。

它真正做的是：

> **把第一阶段的产品边界钉死：前台只服务家长，底层开始生长引擎。**

这样你不会出现两种最坏结果：

- 要么第一阶段看起来太像“通用成长平台”，导致首单转化很差
- 要么第一阶段做成一堆教育专用逻辑，导致第二阶段全部推倒

这份文档的使命，就是让第一阶段既能卖，又不白做。
