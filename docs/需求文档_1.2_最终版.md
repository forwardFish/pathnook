# 3. 专业诊断主线：六步流水线（必须统一理解）

本节是整份文档最重要的总纲。  
后续所有模块，不要分开理解，要按这条流水线理解：

```text
先收集诊断上下文输入（Diagnostic Intake）
-> 再把上传材料拆成结构化问题项（Problem Item）
-> 再做错误分类（Taxonomy）
-> 再聚合成主诊断 / 次诊断 / 模式问题（Diagnosis）
-> 再生成 7 天行动处方（7-Day Plan）
-> 最后用置信度做门禁（Confidence Gate）
```

## 3.1 为什么必须按流水线做
因为系统不能直接看一堆图片，然后让 LLM “自由发挥写报告”。

那样会导致：
- 报告漂
- 每次风格不同
- evidence 对不回去
- plan 很空
- 低质量输入也会被包装成“很专业”

所以必须先把系统变成：
- **事实层**
- **分类层**
- **判断层**
- **处方层**
- **质量层**

## 3.2 六步分别在做什么

### 第一步：Diagnostic Intake
作用：定义“这次在看什么病”。  
目的：让系统知道这次到底是考后分析、作业反复出错、请 tutor 前预诊断，还是周复盘。

### 第二步：Problem Item
作用：把图片/PDF 变成一题一题的结构化事实。  
目的：没有 `problem_items`，就没有 evidence。

### 第三步：Taxonomy
作用：给每道题贴错误标签。  
目的：统一系统内部的“错误语言”。

### 第四步：Diagnosis
作用：从很多错误标签中聚合出主问题、次问题、模式问题。  
目的：从“错了哪些题”升级到“到底出了什么问题”。

### 第五步：7-Day Plan
作用：根据诊断结果开一周处方。  
目的：让家长知道这一周具体怎么做。

### 第六步：Confidence Gate
作用：判断这份报告到底该不该发、怎么发。  
目的：防止低质量输入被包装成高质量报告。

## 3.3 这六步解决的不是“功能”，而是“专业感”
如果只做页面和报告壳子，系统会像一个 AI summarizer。  
只有把六步都做完整，系统才会像：

> **一个会看材料、会做判断、会开处方、知道什么时候不该乱说的诊断系统。**

# 4.2 5.2 Problem Item：事实层抽取

## 核心目标
把上传页材料拆成最小分析单位：`problem_items`。  
这一层相当于病历里的“检查结果记录”。

## 为什么必须有
没有 `problem_items`，后续就没法做：
- evidence
- taxonomy
- diagnosis scoring
- page/prob anchor
- tutor-readable explanation

## 最小结构
```json
{
  "pageNumber": 2,
  "problemNumber": "Q4",
  "questionSummary": "Fraction addition with unlike denominators",
  "studentResponseSummary": "Added numerators directly without finding a common denominator",
  "teacherMark": "wrong",
  "confidence": 0.82
}
```

## 推荐增强字段
- `sourceSnippet`
- `evidenceAnchor`
- `workType`
- `topicHint`
- `normalizedTeacherSignal`

## 字段解释
### `questionSummary`
不是 OCR 原文全文，而是题型摘要。  
作用：
- 后续分类
- report 展示
- tutor summary

### `studentResponseSummary`
不是“这题错了”，而是：
- 学生做了什么
- 错在什么动作
- 是否缺步骤 / 漏符号 / 跳步骤

### `teacherMark`
枚举：
- `correct`
- `wrong`
- `partial`
- `unknown`

### `workType`
- `multiple_choice`
- `computation`
- `open_response`
- `word_problem`
- `geometry`
- `mixed`

### `topicHint`
- `fractions`
- `equations`
- `geometry`
- `arithmetic`
- `word_problems`
- `unknown`

## 数据库建议
在现有 `problem_items` 上新增：
- `questionSummary`
- `studentResponseSummary`
- `workType`
- `topicHint`
- `sourceSnippet`
- `evidenceAnchorJson`
- `normalizedTeacherSignal`

## API
- `POST /api/runs/[runId]/extract-items`
- `GET /api/runs/[runId]/items`

## 页面关系
### 运行状态页
必须显示：
- extracting items
- items extracted
- low extraction confidence warning

### 报告页
Evidence 中每个 finding 都必须可回到 problem item

## 验收标准
- 每个 report 都有可用 items
- `Top Findings` 至少能追溯到 1 个 item
- item 能追溯到 page / anchor
- OCR 差时不伪造 items

# 4.3 5.3 Taxonomy：错误分类层

## 核心目标
给每个 `problem_item` 贴错误标签，形成统一错误语言。  
这一层相当于“症状编码系统”。

## 为什么它是核心壁垒
因为：
- 没有 taxonomy，就没有稳定 diagnosis
- 没有 taxonomy，就没有可对比趋势
- 没有 taxonomy，就没有长期学习画像

## MVP taxonomy（固定 8 类）
1. `concept_gap`
2. `procedure_gap`
3. `calculation_slip`
4. `reading_issue`
5. `notation_error`
6. `strategy_error`
7. `careless_slip`
8. `incomplete_reasoning`

## 每个 item 的输出结构
```json
{
  "errorType": "procedure_gap",
  "severity": "high",
  "confidence": 0.76,
  "rationale": "Student skipped the common denominator step before adding fractions."
}
```

## 严重程度规则
- `low`
- `medium`
- `high`

建议：
- `low`：单题轻度偏差
- `medium`：明显错误，但未必构成模式
- `high`：系统性错误，可能影响后续学习

## 多标签策略
MVP 建议：
- 每题最多 2 个标签
- 1 个 `primaryErrorType`
- 1 个可选 `secondaryErrorType`

## 数据库建议
沿用 `item_errors`，但必须有：
- `problemItemId`
- `errorType`
- `severity`
- `confidence`
- `rationale`
- `isPrimary`

## 关键约束
- 每个 Top Finding 必须至少有 taxonomy 支撑
- 无 rationale 的 label 不可进入 report
- 低 confidence label 不可直接进入 primary diagnosis 聚合

## 验收标准
- report 中的 Top Findings 都有 taxonomy 支撑
- rationale 能回到 evidence
- 多页同类错误可稳定聚合成 pattern issue

# 4.4 5.4 Diagnosis 聚合：从错误到判断

## 核心目标
把很多问题项和 taxonomy 标签，聚合成真正的诊断结论。  
这一层相当于“病名判断层”。

## 这一层必须回答的 5 个问题
1. 主要问题是什么
2. 次问题是什么
3. 哪些是模式性问题
4. 哪些只是偶发
5. 哪些现在不必过度反应

## 本模块必须输出
- `primaryDiagnosis`
- `secondaryDiagnosis`
- `patternIssues`
- `sporadicIssues`
- `doNotOverreactPoints`

## Diagnosis 不是“标签计数器”
不能简单按 taxonomy 次数排序。  
必须引入：

> **issue tree + hypothesis scoring**

## Diagnosis scoring 规则
```text
Diagnosis Score =
(error frequency * 0.4)
+ (severity weight * 0.3)
+ (cross-page repetition * 0.2)
+ (teacher signal support * 0.1)
```

## 关键判断逻辑
### Pattern issue
满足任一：
- 同 taxonomy 在 2 页以上出现
- 同 taxonomy 出现 >= 3 次
- 与上一份报告同类型问题重复出现

### Sporadic issue
满足：
- 只出现 1 次
- 不构成重复模式
- 不应成为本周主重点

### Do-not-overreact
系统必须主动告诉家长：
- 哪些问题只是偶发
- 哪些暂时不用升级处理
- 不要把所有错都当成“基础全盘不行”

## 数据库存储建议
在 `reports` 上新增/规范：
- `primaryDiagnosisCode`
- `secondaryDiagnosisCodesJson`
- `patternSummaryJson`
- `sporadicSummaryJson`
- `doNotOverreactJson`
- `overallConfidence`

## 页面改造
页面：`/dashboard/reports/[reportId]`

新增区块：
- `Primary Diagnosis`
- `Secondary Diagnosis`
- `Pattern Issues`
- `Sporadic Issues`
- `Do not overreact to`

## 用户体验目标
家长看完后能复述：
- 主要问题是什么
- 为什么是这个问题
- 哪些先别慌
- 本周先抓什么

## 验收标准
- 报告不再只是罗列错误
- report 首屏更像“判断书”
- 判断和 evidence 一一对应
- 可明确区分 pattern / sporadic

# 4.5 5.5 7-Day Plan：处方层

## 核心目标
根据 diagnosis，给出一周可执行处方。  
这一层相当于“治疗方案层”。

## 为什么它决定用户值不值
家长不是为“分析”买单，  
而是为：

> **分析之后，知道这一周怎么做**  
> **以及这一周不要做什么**

## 原则
不能只靠 LLM 自由生成。  
必须采用：

> **规则模板库 + LLM 补充表达**

## 每天输出结构
```json
{
  "day": 1,
  "minutes": 20,
  "goal": "Rebuild fraction step order",
  "tasks": [
    "Review how to find a common denominator",
    "Complete 5 structured fraction problems"
  ],
  "parentPrompt": "Show me how you found the common denominator.",
  "successCheck": "Can explain the full step order without help"
}
```

## 必须字段
- `day`
- `minutes`
- `goal`
- `tasks`
- `parentPrompt`
- `successCheck`

整周必须有：
- `pauseList`

## pauseList 为什么必须有
这一步很值钱，因为它能明确告诉家长：

- 这周不要加大题量
- 这周不要上更难混合题
- 这周不要频繁换方法
- 这周不要把偶发粗心当主问题

## 规则库设计建议
新增 `plan_rule_templates`

### procedure_gap
- Day 1–2：慢速步骤重建
- Day 3–4：同类短练
- Day 5：轻微迁移
- Day 6：错题回看
- Day 7：mini-check

### concept_gap
- Day 1：概念重讲 / 自己解释
- Day 2–3：概念题 + 基础题
- Day 4–5：概念到步骤连接
- Day 6：轻量混合短练
- Day 7：回顾与小测

### reading_issue
- Day 1：题干拆解
- Day 2：条件标记
- Day 3：数量关系翻译
- Day 4–5：同类 word problem
- Day 6：错题复盘
- Day 7：mini-check

## 数据库建议
`reports.planJson` 继续使用，但新增：
- `planVersion`
- `planRuleCodesJson`

## 页面改造
`7-Day Plan` tab 以 day card 展示：
- Day number
- minutes
- goal
- tasks
- parent prompt
- success check

## 验收标准
- plan 必须和 diagnosis 对应
- plan 必须具体
- 家长本周能执行
- 同一个 diagnosis 的 plan 不应完全复制粘贴

# 4.6 5.6 Confidence Gate：质量门禁与 Review Gate

## 核心目标
决定这份报告到底该不该发、怎么发。  
这一层相当于“出报告前的质控层”。

## 为什么一定要做
如果不做 confidence gate，会出现两个严重问题：

1. 低质量输入也输出很肯定的结论
2. 报告看起来像真的，但其实是乱判

## 三档质量层级
### A 档：高置信度
条件：
- extraction 稳定
- evidence 足够
- taxonomy 聚合稳定
- diagnosis clear

处理：
- 自动发布正式版报告

### B 档：中置信度
条件：
- extraction 可用，但不够稳
- 部分页模糊或信息不完整
- diagnosis 有倾向，但不够强

处理：
- 发布谨慎版报告
- 提示：
  - some conclusions may need clearer pages
  - consider uploading correction pages

### C 档：低置信度
条件：
- OCR 很差
- problem items 太少
- evidence 不足
- diagnosis 冲突大

处理：
- 进入 `needs_review`
- 不直接展示完整报告
- 引导补拍 / 补充订正页 / 等待审核

## 诊断质量门禁（Diagnosis Quality Gate）
新增统一评分表。

### 一级门禁（必须全过）
1. 是否有 `primaryDiagnosis`
2. 是否有 `pattern vs sporadic`
3. 每个 Top Finding 是否至少有 2 条 evidence
4. 是否有 `this week focus`
5. 是否有 `pauseList`
6. 7-Day Plan 是否每天可执行

### 二级门禁（建议项）
1. summary 是否因 intake 场景发生差异
2. tutor share 是否比 parent 版更专业
3. 报告措辞是否避免模板化
4. 是否明确说明“不必过度反应”的点

## 质量等级建议
- `A`：可直接发布
- `B`：可发布，但建议抽检
- `C`：needs_review
- `D`：不允许发布

## 数据库建议
### analysis_runs
新增：
- `qualityTier`
- `needsReviewReason`
- `retryCount`
- `userVisibleStatus`

### reports
新增：
- `qualityGateScore`
- `qualityFlag`
- `reviewRequired`

## 页面改造
### 运行状态页 `/dashboard/runs/[runId]`
增加阶段可视化：
- extracting items
- classifying errors
- building diagnosis
- generating plan
- quality check

### 报告页
如果是 B 档，显示：
- `Cautious report`
- `Some conclusions may become stronger with clearer pages`

如果是 C 档：
- 不进入正式 report
- 引导 `needs_review` / 补拍

## 验收标准
- 不允许无 evidence 强结论
- 不允许低质量输入伪装成高质量输出
- 用户能理解为什么被要求补拍 / 复审

# 5. 专业诊断不等于只做完这 6 步：还必须有校准层

这部分是我们刚才讨论后，新增进最终版的重要章节。

## 5.1 为什么需要校准层
做完六步后，你会得到：

> **结构上专业的诊断系统**

但还不自动等于：

> **判断上专业、商业上可信、用户愿意持续付费的诊断系统**

因此，必须补一个“校准层”。

## 5.2 校准层包含什么

### A. 黄金样本库（Gold Set）
建立 20–30 份真实上传材料样本。  
每份样本人工先做出：
- Diagnostic Intake
- Problem Items
- Taxonomy labels
- Primary / Secondary Diagnosis
- Pattern / Sporadic 判断
- Do-not-overreact 点
- 7-Day Plan

这些样本用于：
- few-shot prompt
- taxonomy 校准
- diagnosis 校准
- 质量评分基准

### B. Taxonomy 判定手册
每个 taxonomy 至少定义：
- 正例
- 反例
- 边界例
- 不应误判成什么

### C. Diagnosis 聚合规则
必须写明：
- 主诊断优先解释最多错误模式
- 不只是“哪个 taxonomy 数量最多”
- 表层标签和根因标签如何区分

### D. Plan 模板库
至少先做 MVP 对应的 rule templates：
- `procedure_gap`
- `concept_gap`
- `reading_issue`
- `notation_error`
- `incomplete_reasoning`

### E. 抽检机制
建议每周抽检：
- 新报告 10%
- 所有 `qualityTier = B`
- 所有被用户反馈的报告

## 5.3 为什么这层很重要
没有校准层，系统会：
- 今天判断 procedure_gap
- 明天判断 concept_gap
- 后天又给出完全不同 plan

这样报告“看起来专业”，但不稳定。

## 5.4 本版本是否必须做完校准层
### 本版本必须至少做：
- 黄金样本库起步版（>=10份）
- taxonomy 判定手册起步版
- diagnosis 聚合规则文档版
- 7-Day Plan 模板库起步版

### 本版本可以暂缓：
- 大规模抽检后台
- 复杂人审运营系统
