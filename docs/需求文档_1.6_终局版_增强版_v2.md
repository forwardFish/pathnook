# B. 高手学习系统总纲

## B.1 Pathnook 的终局，不是“更聪明的教育工具”
它不应该停留在：
- AI 讲题
- AI 批改
- AI 报告
- AI 计划
- AI 打卡

这些能力都可以有，但它们只是外层形态。

Pathnook 终局真正要做的是：

> **把高手的学习方式，工业化、系统化、普惠化。**

换句话说：

- 过去高手之所以快，是因为他们更会判断
- 更会抓本质
- 更会找杠杆
- 更早输出
- 更快纠偏
- 更善于持续迭代

而 Pathnook 想做的，是通过 AI 把这些能力外化成系统，让普通人也能借助系统，按照更接近高手的方式学习。

---

## B.2 终局产品的一句话定义
> **Pathnook is an AI learning and growth system that helps ordinary users learn in a way that is closer to how experts learn.**

中文对应：

> **Pathnook 是一个 AI 学习成长系统，它的终局不是帮用户多学一点，而是帮普通人以更接近高手的方式学习。**

---

## B.3 终局产品的核心使命
Pathnook 不是单纯帮助用户：
- 看懂更多内容
- 做更多题
- 打更多卡

而是帮助用户：

1. 更快抓住骨架  
2. 更快发现杠杆点  
3. 更快进入真实输出  
4. 更快获得高密度反馈  
5. 更快修正偏差  
6. 更持续地积累成长轨迹  
7. 最终形成自己的自我纠错系统  

---

# C. 高手学习方式模型

> 本节不是抽象讨论，而是后续所有产品功能设计的顶层约束。

## C.1 高手不是在学内容，而是在抓骨架
高手优先问：
- 这门东西的骨架是什么
- 最关键的 20% 是什么
- 哪些概念一旦打通，后面会连起来
- 哪些内容当前先不用学

### 产品翻译
Pathnook 必须具备：
- 骨架识别
- 杠杆点判断
- 忽略项判断

---

## C.2 高手不是按教材顺序学，而是按能力生成顺序学
高手优先问：
- 什么能力最先需要长出来
- 什么概念最能带动后面
- 什么任务最能暴露问题
- 什么练习最能快速形成反馈

### 产品翻译
Pathnook 必须具备：
- 最短能力路径
- 阶段目标
- 当前优先级排序

---

## C.3 高手始终围绕输出学习
高手会很早进入输出：
- 讲给别人听
- 做题
- 写代码
- 做项目
- 拆案例
- 重建过程
- 做变式

### 产品翻译
Pathnook 必须具备：
- 输出关卡
- 输出任务类型库
- 输出结果判定
- 通过标准

---

## C.4 高手不是机械重复，而是高频纠偏
高手成长快，不是因为练得多，而是因为：
- 暴露问题更早
- 得到反馈更快
- 修正错误更快

### 产品翻译
Pathnook 必须具备：
- 高频纠偏反馈
- 重复错误识别
- 当前纠偏目标
- 下次修正动作

---

## C.5 高手不断建立判断力
高手最强的不是知识量，而是判断力：

- 什么重要
- 什么不重要
- 什么时候继续
- 什么时候换方法
- 什么错误值得重视
- 什么问题只是波动

### 产品翻译
Pathnook 必须具备：
- 主因 / 次因判断
- pattern / sporadic 区分
- ignore-for-now
- leverage-first

---

## C.6 高手不断建立自我纠错系统
高手最终不是靠外部盯，而是自己体内长出一套系统：

- 发现偏差
- 识别问题
- 判断轻重缓急
- 选择下一步
- 持续修正

### 产品翻译
Pathnook 终局必须帮助用户逐渐形成：
- 自我判断
- 自我复盘
- 自我续接
- 自我迭代

也就是说，Pathnook 不只是代替判断，更要训练用户形成判断。

# D. 产品能力翻译层（高手方式 → 产品模块）

## D.1 骨架地图系统（Skeleton Map System）

- 当前骨架节点的含义
- 当前最值得优先处理的关键 20%

### 终局功能

- Current node
- Leverage focus
- Ignore-for-now

### MVP 对应轻版

### 作用

- 哪个能力节点一旦突破，会带动最多结果

### 终局功能

- dependency graph
- bottleneck reasoning
- secondary effect estimation

- Secondary issue
- This week’s focus
- Most valuable next fix

## D.3 最短能力路径系统（Shortest Ability Path System）
### 作用
帮助用户不是“看教材目录”，而是看到：
- 当前最短可行路径
- 为什么先走这一步
- 这一阶段的唯一目标
- 下一步是什么

### 终局功能

- entry condition

# E. MVP 到终局的演进逻辑

## E.1 MVP 不是终局缩小版，而是终局的最小证明
MVP 不可能一开始就做完高手学习系统。  
MVP 只需要证明两件事：

1. 用户会不会明显感觉到“这不是普通 AI 工具”
2. 用户会不会因为这种差异化价值而付费并回来

所以 MVP 只抓两个突破点最合理：

### 突破点 1：骨架诊断
让用户明确感觉：
> 系统抓到的不是表面错题，而是真正卡点。

### 突破点 2：最短能力路径 + 输出关卡
让用户明确感觉：
> 系统不是让我继续看，而是让我用最短路径暴露真实水平并修正。

---

## E.2 Stage 1 是闭环版
Stage 1 的核心不是再多讲理念，而是把：

- 判断
- 行动
- 反馈
- 续接

做成稳定闭环。

这部分就是原版 `1.6_终局版` 已经写得很完整的内容，后文完整保留。

---

## E.3 终局不是“更多页面”，而是“更完整的高手学习系统”
终局产品的衡量标准，不是：
- 页面更多
- 图表更多
- 互动更多

而是：
- 是否更快抓到骨架
- 是否更快给出最短路径
- 是否更快逼出真实输出
- 是否更快纠偏
- 是否更强地支持持续迭代
- 是否逐渐帮助用户长出自己的判断力

在保留原版所有工程细节的前提下，再增加 6 条原则：

1. 不要把终局产品理解成“更复杂的 report 系统”

# 1. 设计总原则（给 Codex 的最高约束）

## 1.1 产品定位约束

从 `1.6` 开始，所有开发都必须同时满足：

1. **Pathnook 的大定位是 AI 学习成长系统**
2. **当前前台必须仍然是家庭教育学习产品**
3. **默认用户仍然是家长，不是学生**
4. **Diagnosis / Evidence / 7-Day Plan 仍然是主体验**
5. **Walkthrough 仍然是增强层，不得反客为主**

---

## 1.2 架构约束

1. **不推翻重做当前系统**
2. **继续沿用当前 Next.js + Vercel 一体化架构**
3. **继续沿用当前 Parent Dashboard / Child / Upload / Run / Report / Share / Billing 的主路径**
4. **所有新增能力必须优先复用当前页面和领域对象**
5. **任何新增重能力都要先判断：是否真的帮助“判断—行动—反馈—续接”闭环**

---

## 1.3 开发约束

### 强制规则
- 不允许把所有生成结果继续压成一个大 `reportJson blob`
- 不允许跳过结构化层，直接让模型自由发挥
- 不允许为了“更像课堂”破坏 parent-first report 体验
- 不允许在首页、pricing、sample-report、report 主体验里把 Walkthrough 变成主角
- 不允许为了以后扩展，把 Stage 1 当前转化所需的清晰度做模糊

### 必须坚持的对象层
以下对象必须继续作为核心对象，而不是临时字段：
- `ProblemItem`
- `EvidenceAnchor`
- `ItemErrorLabel`
- `DiagnosisOutline`
- `ExplanationCard`
- `SevenDayPlan`
- `QualityGateResult`
- `ShareArtifact`
- `ReviewHistory / IssueTrend`
- `PlaybackSnapshot`（轻版）

# 2. Stage 1 的产品闭环定义

## 2.1 最小闭环主流程

```text
Landing / Pricing / Sample Report
-> Sign up / Login
-> Add child
-> Upload recent learning evidence
-> Complete diagnostic intake
-> Run created
-> Parsing / ProblemItem extraction / Taxonomy / Diagnosis
-> Quality Gate
-> Billing unlock (if applicable)
-> Report: Diagnosis / Evidence / 7-Day Plan / Share
-> Weekly Review / Compare (light)
-> Re-upload next week
-> Resume from prior state
```

---

## 2.2 四大系统在 Stage 1 的具体落点

### 学习判断系统 → Report 首屏
产品落点：
- Diagnosis 区
- Evidence 区
- What not to overreact to
- This week’s focus

3. 7-day plan 必须可执行
4. report 不依赖 walkthrough 也能成立
5. tutor summary 可分享、可理解
6. compare with last report 至少能看出变化方向
7. child detail 能显示 current focus 和 next best action
8. 中断回来后，用户不会不知道接下来做什么

## 10.2 工程验收

1. 数据不再只躺在 report blob 里
2. 核心对象真实落表
3. 低质量 run 不自动正式发布
4. share token / revoke / expiry 稳定
5. billing 解锁与 report 访问状态正确
6. walkthrough 轻版可手动进入、可暂停、可恢复
7. smoke 流程覆盖 upload -> run -> report -> share -> billing

## 10.3 边界验收

以下情况出现，视为偏离 1.6：

- 首页重新讲成数学工具或双语工具
- 报告页被播放器主导
- 先做大而全趋势系统，却没把 diagnosis / plan 做稳
- 为了“以后扩展”牺牲当前家长转化
- 把 student app、team SaaS、adult growth 混入当前主排期
