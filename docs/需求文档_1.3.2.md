# 0. 执行摘要

## 0.1 这份文档要解决什么问题

你当前系统已经具备：

- 上传作业 / PDF
- 生成学习诊断报告
- 展示 Evidence / 7-Day Plan
- PDF Export
- Tutor Share
- Admin Review
- 周度复盘方向

但当前系统的核心输出仍然偏“静态报告”。

这会带来 4 个问题：

1. **报告可读，但不够“会讲”**  
   家长能看结论，但很难顺着系统一步步理解“为什么是这个问题”。

2. **错题讲解有内容，但缺乏统一交互载体**  
   现在的讲解更像报告中的一段文字，而不是可回放、可暂停、可分享的“讲解过程”。

3. **Tutor 分享是静态沟通材料，不是动态沟通工具**  
   发给 tutor 后，对方只能看摘要，不能像看一个“会演示的诊断 deck”那样快速进入状态。

4. **PDF 可以导出，但不是最强体验**  
   真正高感知价值不是“多一个 PDF”，而是“系统像一个会讲解的专业教练，带着家长一步步看懂问题”。

因此，`1.3.2` 的目标是新增一个中间层：

> **Diagnosis Player Layer（学习诊断报告播放器层）**

它把你已有的：

- Diagnosis（判断）
- Evidence（证据）
- Explanation（讲解）
- 7-Day Plan（处方）

统一编排成：

> **Diagnosis Deck → Diagnosis Slide → Explanation Action → Playback**

最终实现一个不是 PPT、但拥有“类 PPT 展示 + 语音播报 + 高亮证据 + 白板讲解 + H5 分享”体验的系统。

---

## 0.2 本版本的最终交付效果

本版本上线后，单个报告除了原来的 `/report` 静态阅读页，还会新增一个 `/play` 播放页。

用户看到的不是一篇长文，而是一份**可播放的诊断讲解 deck**。

### 家长端最终体验

家长上传 5–10 页作业/试卷后，系统除生成静态报告外，还自动生成一个“诊断讲解播放器”：

- 第 1 页：封面页 / 本次主题
- 第 2 页：本次主诊断（Primary Diagnosis）
- 第 3 页：模式性问题 vs 偶发问题
- 第 4 页：证据页（哪几道题支撑结论）
- 第 5–7 页：逐题讲解页 / 按模式讲解页
- 第 8 页：7 天训练方案
- 第 9 页：家长本周只盯什么

播放时系统支持：

- 自动朗读当前页讲稿
- 高亮题目区域
- 激光笔/聚光灯指向具体证据
- 白板文字 / 数学公式演示
- 暂停 / 继续 / 下一页 / 上一页 / 重新播放

### Tutor 端最终体验

家长通过分享链接发给 tutor 后，对方可以：

- 打开只读播放页
- 快速看摘要
- 直接播放讲解卡片
- 跳到证据页看题目原图
- 看 7 天训练重点

### 内部运营端最终体验

当 run 质量不足时：

- 播放版不自动对外开放
- 进入 `needs_review`
- 管理员可先审查 slide 内容、讲解卡片、动作脚本
- 审核后再发布

---

## 0.3 本版本不做什么

为了保证能直接开发、可控落地，本版本明确不做：

- 不做 OpenMAIC 那种完整多智能体课堂系统
- 不做“学生和 AI 实时来回对话”的流式教学系统
- 不做复杂虚拟人 / 头像驱动 / 视频生成
- 不做复杂 PBL / 小组讨论 / 多角色课堂
- 不做完整 PPT 编辑器
- 不做导师端 SaaS 独立产品
- 不做原生 App
- 不做多学科扩张

本版本只做：

> **静态诊断报告之上，增加“可播放的诊断讲解体验层”**

# 2. 产品目标与用户价值

## 2.1 产品目标

本版本要达成的不是“技术上能播放”，而是 3 个产品目标：

### 目标 A：让诊断报告更像“会讲的教练”

报告不再只是结论和文字，而是一步步带家长理解：

- 这次核心问题是什么
- 为什么是这个问题
- 哪些题最能说明问题
- 本周该怎么做

### 目标 B：让 Tutor 分享更有价值

Tutor 不再只能看文字摘要，而是能快速通过播放页进入上下文。

### 目标 C：为未来扩展打底

未来无论你要扩：

- 更强的错题讲解
- 几何辅助线演示
- 语音更自然
- H5 课程包
- 导师讲解稿导出

都基于同一个 Player Layer 继续扩，而不是重做一套系统。

---

## 2.2 用户价值

### 对家长的价值

- 看得懂：不是“AI 生成一篇看不下去的文档”
- 更信：结论和证据一起被讲出来
- 更易执行：7 天计划不再是静态清单，而是能讲清“为什么先抓这个”

### 对 tutor 的价值

- 上手快：打开就能播放，不必先读完长报告
- 沟通省时间：证据、重点、建议都在同一个讲解流程里

### 对你自己的价值

- 提高产品感知价值
- 给 299 / 999 / 更高服务层级提供体验支撑
- 后续可衍生为 H5 交付、演示型交付、课程化交付

# 3. 本版本的核心定义

## 3.1 什么是 Diagnosis Deck

Diagnosis Deck 是一份“可播放的诊断讲解包”，是本版本最核心的交付对象。

它不是 PPT 文件，但具备类似特征：

- 有顺序页
- 每页有标题/内容/证据/讲稿
- 每页有动作
- 可以逐页播放
- 可以导出为 H5 或未来导出为 PPT/PDF

### 定义

```ts
export interface DiagnosisDeck {
  id: string;
  reportId: string;
  childId: string;
  runId: string;
  locale: "en-US" | "es-ES" | "zh-CN";
  title: string;
  subtitle?: string;
  version: string;
  status: "draft" | "ready" | "published" | "archived";
  slides: DiagnosisSlide[];
  summary: DeckSummary;
  createdAt: string;
  updatedAt: string;
}
```

---

## 3.2 什么是 Diagnosis Slide

Diagnosis Slide 是 deck 中的一页。

每页既有静态内容，也有“讲解行为”。

### 必备字段

```ts
export interface DiagnosisSlide {
  id: string;
  deckId: string;
  orderIndex: number;
  type:
    | "cover"
    | "primary_diagnosis"
    | "secondary_diagnosis"
    | "pattern_summary"
    | "sporadic_summary"
    | "evidence"
    | "explanation"
    | "plan"
    | "parent_summary"
    | "tutor_summary";
  title: string;
  objective: string;
  layout: SlideLayoutType;
  content: SlideContent;
  speakerNotes?: string;
  actions?: ExplainAction[];
  evidenceRefs?: string[];
  estimatedDurationSec?: number;
}
```

---

## 3.3 什么是 Explanation Card

Explanation Card 是“错题讲解/模式讲解”的原子单元。

它既可以单独显示，也可以被拼进某个 slide 中。

它必须具备：

- 讲解目标
- 证据锚点
- 解释文本
- 步骤说明
- quick check
- 可选动作脚本

---

## 3.4 什么是 Explain Action

Explain Action 是播放器执行的最小动作。

它不是 UI 上的随机事件，而是生成系统明确产出的“可执行讲解动作”。

例如：

- 高亮某题
- 激光笔指向某一步
- 播放讲稿
- 打开白板
- 写公式
- 清空白板
- 切下一页

这部分和 OpenMAIC 的 Action 思想一致，但你这里的 action target 不是 scene element，而是作业证据锚点。

---

## 3.5 什么是 Playback

Playback 指播放器按顺序消费 slide.actions 的过程。

它必须支持：

- 开始播放
- 暂停
- 恢复
- 停止
- 跳到指定页
- 单步执行
- 保存进度
- 恢复进度

# 4. 用户场景与页面流程

## 4.1 场景 1：家长看完整的诊断讲解

### 流程

1. 家长上传作业并生成报告
2. 系统完成 diagnosis / evidence / plan
3. 系统额外生成 diagnosis deck
4. 家长在报告页点击 `Play diagnosis`
5. 进入 `/dashboard/reports/[reportId]/play`
6. 播放器按页展示并自动朗读
7. 家长暂停，在证据区查看原题
8. 播放结束后跳到本周计划页

### 用户价值

- 不需要自己读长文
- 更容易理解“为什么先抓这个”

---

## 4.2 场景 2：家长发给 tutor

### 流程

1. 家长生成 share link
2. tutor 打开 `/share/[token]/play`
3. 默认看到 deck 的第一页摘要
4. tutor 可直接播放全部或跳到 explanation slides
5. tutor 看完后回到静态 report tab

### 用户价值

- 减少 tutor 理解上下文的时间
- 提升“这是一个专业工具”的感知

---

## 4.3 场景 3：低质量报告不开放播放

### 流程

1. OCR 差 / 证据不足 / diagnosis 不稳
2. Quality Gate 评为 C 或 D
3. deck 状态为 `draft` 或 `blocked`
4. report 页只显示静态谨慎版
5. play 按钮隐藏或显示 “Unavailable while reviewing”

### 原则

播放器不是炫技层。  
如果底层诊断不稳，宁可不播，也不要播错。

# 5. 页面与路由设计

## 5.1 新增页面列表

本版本新增以下页面：

- `/dashboard/reports/[reportId]/play`
- `/share/[token]/play`
- `/admin/review/[runId]/deck`

可选新增（内部调试）：

- `/dev/decks/[deckId]`
- `/dev/cards/[cardId]`

---

## 5.2 `/dashboard/reports/[reportId]/play`

### 页面目标

给家长提供完整播放体验。

### 页面布局

```text
+--------------------------------------------------------------------------------+
| Header: Back to report | Deck title | Share | Export | More                   |
|--------------------------------------------------------------------------------|
| Left: Slide canvas                          | Right: Evidence / Notes panel    |
|                                             |                                  |
|   [title]                                   |   Current slide objective        |
|   [main content]                            |   Evidence refs list             |
|   [charts / cards / image snippets]         |   Related page thumbnails        |
|                                             |   Quick jump buttons             |
|--------------------------------------------------------------------------------|
| Bottom controls: Prev | Play/Pause | Next | Progress | Speed | Voice | Replay |
+--------------------------------------------------------------------------------+
```

### 必备组件

- `PlayerHeader`
- `SlideCanvas`
- `EvidencePanel`
- `PlaybackControls`
- `VoiceToggle`
- `ProgressBar`
- `SlideNavigator`

### 行为要求

- 首次进入默认停在第一页，不自动播放（避免打断）
- 点击 Play 后从当前 slide 当前 action 开始
- 右侧面板始终显示当前 slide 对应的 evidence refs
- 用户可直接点击某个 evidence 跳去看原页

---

## 5.3 `/share/[token]/play`

### 页面目标

给 tutor/外部只读用户使用。

### 差异点

- 不显示家长内部备注
- 不显示内部质量标签
- 不显示管理员调试按钮
- 可限制某些 slide 不展示（例如 parent-only summary）

### 额外要求

- token 失效时展示友好页面
- revoked 时明确提示 link no longer available

---

## 5.4 `/admin/review/[runId]/deck`

### 页面目标

让管理员在发布前检查播放版质量。

### 页面必须显示

- deck 元数据
- slide 列表
- slide 内容 JSON
- action JSON
- 试播按钮
- 质量评分
- blocking reasons

### 管理动作

- approve deck
- reject deck
- regenerate actions
- regenerate current slide
- mark as parent-safe

# 6. 信息架构与 slide 结构规范

## 6.1 Deck 的默认页序

V1 固定为 8–9 页，不允许模型自由扩展页数。

### 默认顺序

1. Cover
2. Primary Diagnosis
3. Pattern vs Sporadic
4. Evidence Summary
5. Explanation Slide A
6. Explanation Slide B
7. 7-Day Plan
8. Parent Summary
9. Tutor Summary（可选，仅 tutor share 显示）

### 为什么固定页数

这样可以保证：

- 更稳定
- 更易测
- 更易导出
- 不会出现有的报告 4 页、有的 18 页的混乱状态

---

## 6.2 Slide 类型定义

### 6.2.1 Cover

用途：

- 报告标题
- 学生昵称 / 年级
- 本次资料页数
- 日期

必备字段：

- `title`
- `subtitle`
- `childSummary`
- `materialsSummary`

---

### 6.2.2 Primary Diagnosis

用途：

- 告诉家长本次最该先抓的问题

必备字段：

- `primaryIssueTitle`
- `primaryIssueWhy`
- `impactSummary`
- `priorityRecommendation`

---

### 6.2.3 Pattern Summary

用途：

- 区分模式性问题和偶发问题

必备字段：

- `patternIssues[]`
- `sporadicIssues[]`
- `doNotOverreact[]`

---

### 6.2.4 Evidence Summary

用途：

- 把结论和题目连接起来

必备字段：

- `evidenceGroups[]`
- `pageReferences[]`
- `thumbnails[]`

---

### 6.2.5 Explanation Slide

用途：

- 对某道代表性错题或某种模式做讲解

必备字段：

- `questionSummary`
- `studentMistakeSummary`
- `correctPathSteps[]`
- `commonPitfall`
- `quickCheck`

---

### 6.2.6 Plan Slide

用途：

- 讲清 7 天计划

必备字段：

- `days[]`
- `pauseList[]`
- `focusSummary`

---

### 6.2.7 Parent Summary

用途：

- 告诉家长本周只盯什么

必备字段：

- `oneThingToWatch`
- `whatToSayToChild`
- `whatNotToDo`
- `nextUploadReminder`

# 7. 数据模型设计

## 7.1 Deck 数据结构

```ts
export interface DeckSummary {
  slideCount: number;
  totalEstimatedDurationSec: number;
  explanationCardCount: number;
  hasAudio: boolean;
  hasWhiteboard: boolean;
}

export type SlideLayoutType =
  | "cover_centered"
  | "two_column"
  | "evidence_focus"
  | "card_focus"
  | "plan_grid"
  | "summary_bullets";

export interface SlideContent {
  blocks: SlideBlock[];
}

export type SlideBlock =
  | { type: "heading"; text: string }
  | { type: "paragraph"; text: string }
  | { type: "bullet_list"; items: string[] }
  | { type: "evidence_list"; items: EvidenceRefBlock[] }
  | { type: "plan_days"; days: PlanDay[] }
  | { type: "explanation_card"; card: ExplanationCard }
  | { type: "callout"; text: string; tone: "info" | "warning" | "success" }
  | { type: "image_thumb"; pageId: string; caption?: string };

export interface EvidenceRefBlock {
  evidenceAnchorId: string;
  label: string;
  pageNumber: number;
  problemNumber?: string;
}
```

---

## 7.2 Action 数据结构

```ts
export interface ActionBase {
  id: string;
  title?: string;
  estimatedDurationMs?: number;
}

export type ExplainAction =
  | ({ type: "speech"; text: string; audioUrl?: string; voice?: string; speed?: number } & ActionBase)
  | ({ type: "spotlight"; anchorId: string; dimOpacity?: number } & ActionBase)
  | ({ type: "laser"; anchorId: string; color?: string } & ActionBase)
  | ({ type: "show_label"; text: string } & ActionBase)
  | ({ type: "wb_open" } & ActionBase)
  | ({ type: "wb_draw_text"; content: string; x: number; y: number } & ActionBase)
  | ({ type: "wb_draw_latex"; latex: string; x: number; y: number } & ActionBase)
  | ({ type: "wb_draw_line"; startX: number; startY: number; endX: number; endY: number } & ActionBase)
  | ({ type: "wb_clear" } & ActionBase)
  | ({ type: "wb_close" } & ActionBase)
  | ({ type: "wait"; ms: number } & ActionBase)
  | ({ type: "next_slide" } & ActionBase);
```

### 约束

- `spotlight/laser` 必须引用当前 slide 可见的 evidence anchor
- `speech.text` 不允许为空
- `next_slide` 只能是 slide 内最后一个动作之一
- `wb_draw_latex` 只允许数学安全子集

# 9. 生成流水线设计

## 9.1 总体流程

本版本新增后的完整流程：

```text
Upload
-> PDF/Image parse
-> Problem Items
-> Evidence Anchors
-> Taxonomy
-> Diagnosis Outline
-> Report Sections
-> Explanation Cards
-> Seven-Day Plan
-> Quality Gate
-> Diagnosis Deck Outline
-> Diagnosis Slides
-> Slide Actions
-> Deck Quality Gate
-> Publish Report + Publish Deck
-> Export PDF / H5
```

---

## 9.2 为什么播放器不能直接从 report 页面拼出来

不能直接把 report 页面 DOM 拿去播放，原因有 4 个：

1. DOM 不是稳定的数据层
2. report 为阅读优化，player 为讲解优化
3. player 需要动作脚本，而 report 没有
4. 未来 H5 / 导出都需要结构化 deck，而不是网页截图

所以必须明确新增：

- `Deck Outline`
- `Slide Content`
- `Slide Actions`

---

## 9.3 Deck Outline 生成

在 diagnosis / explanation / plan 生成完成后，再生成一个 deck outline。

### 输入

- `DiagnosisOutline`
- `ParentReportJson`
- `ExplanationCards[]`
- `SevenDayPlan`
- `QualityGateResult`

### 输出

```ts
export interface DiagnosisDeckOutline {
  title: string;
  slideOutlines: Array<{
    id: string;
    type: DiagnosisSlide["type"];
    title: string;
    objective: string;
    sourceRefs: string[];
  }>;
}
```

### 规则

- 固定 8–9 页
- 解释页不超过 3 页
- 只取 top evidence / top explanation cards
- 若质量不足，可减少 explanation slide 数量

---

## 9.4 Slide 内容生成

每个 slide outline 单独生成 slide content。

### 原则

- 一页一生成
- 一页一校验
- 一页可重试
- 不允许整个 deck 一次生成到底

### 生成顺序

1. cover
2. primary diagnosis
3. pattern summary
4. evidence summary
5. explanation slides
6. plan
7. parent summary

---

## 9.5 Action 脚本生成

每个 slide 单独生成 action 列表。

### 原则

- content 和 actions 分开生成
- actions 必须基于当前 slide 的 content 和 evidence
- actions 只能使用白名单
- actions 生成失败时 slide 仍可展示，只是不能自动播放

### 白名单

V1 允许：

- `speech`
- `spotlight`
- `laser`
- `show_label`
- `wb_open`
- `wb_draw_text`
- `wb_draw_latex`
- `wb_draw_line`
- `wb_clear`
- `wb_close`
- `wait`
- `next_slide`

V1 不允许：

- discussion
- camera move
- avatar animation
- video insert
- external browser

# 10. 播放器设计

## 10.1 播放器状态机

### 状态

- `idle`
- `playing`
- `paused`
- `ended`
- `error`

### 事件

- `START`
- `PAUSE`
- `RESUME`
- `STOP`
- `NEXT_ACTION`
- `NEXT_SLIDE`
- `PREV_SLIDE`
- `RESTORE`
- `AUDIO_ERROR`
- `ACTION_ERROR`

### 状态转移

```text
idle -> START -> playing
playing -> PAUSE -> paused
paused -> RESUME -> playing
playing -> STOP -> idle
playing -> all actions done -> ended
playing -> ACTION_ERROR -> error
error -> START -> playing
```

---

## 10.2 PlaybackEngine 最小实现

```ts
export class PlaybackEngine {
  private state: PlaybackState;
  private deck: DiagnosisDeck;

  constructor(deck: DiagnosisDeck, initialState?: PlaybackState) {
    this.deck = deck;
    this.state = initialState ?? {
      deckId: deck.id,
      slideId: deck.slides[0]?.id ?? "",
      slideIndex: 0,
      actionIndex: 0,
      mode: "idle",
      updatedAt: new Date().toISOString(),
    };
  }

  start(): void {}
  pause(): void {}
  resume(): void {}
  stop(): void {}
  nextAction(): void {}
  nextSlide(): void {}
  prevSlide(): void {}
  restore(snapshot: PlaybackSnapshot): void {}
  getSnapshot(): PlaybackSnapshot { throw new Error("not implemented"); }
}
```

---

## 10.3 ActionEngine 最小实现

ActionEngine 负责执行单个动作。

### 职责

- 接收 `ExplainAction`
- 判断动作类型
- 更新 UI
- 调用 TTS 或音频播放
- 处理失败降级

### 关键要求

- 幂等：重复执行相同 actionId 不产生错误副作用
- 可中断：用户暂停时停止后续动作推进
- 不阻塞：视觉高亮类动作尽量轻量

---

## 10.4 播放策略

### 自动播放模式

- 顺序执行 actions
- `speech` 完成后再走下一个动作
- `wait` 用于给用户视觉停留时间

### 单步模式

- 每点一次 Next 只执行一个 action
- 适合管理员检查动作

### 页模式

- 播完当前页所有 actions 自动切下一页
- 若用户关闭 `auto advance`，则停在页尾

---

# 11. 语音播报方案

## 11.1 V1 默认方案

### 优先级 1：浏览器 TTS

适用原因：

- 开发快
- 无需新增服务端音频存储
- 可先把产品体验跑通

### 行为

- `speech.text` 直接通过浏览器语音 API 播放
- 用户可调语速
- 用户可静音

### 缺点

- 声音一致性差
- 浏览器兼容性不完全一致

---

## 11.2 V1.1 可选增强

### 服务端预生成音频

动作中带：

- `audioUrl`
- `voice`
- `duration`

执行时：

- 优先播 `audioUrl`
- 没有则退回浏览器 TTS

### 适用时机

- 付费用户体验增强
- tutor 分享页更稳定

---

## 11.3 语音失败降级策略

- TTS 不可用：仅显示字幕，不中断播放
- 音频 URL 失效：改用浏览器 TTS
- 浏览器 TTS 被禁：显示“Tap to continue”

# 14. 质量门禁

## 14.1 Deck Quality Gate

播放器版本必须有独立质量门禁。

### 必过项

1. deck 至少有 6 页
2. 每个 explanation slide 至少有 1 个 evidence ref
3. 每个 explanation slide 至少有 1 个 speech action
4. plan slide 必须存在
5. parent summary 必须存在
6. actions 中不允许引用无效 anchor

### 降级项

- 如果 actions 生成失败，但 slide 内容完整：
  - 允许静态展示
  - 不允许 autoplay

- 如果语音生成失败：
  - 允许文字讲稿模式

- 如果 evidence 不足：
  - 允许保留 static report
  - 不生成播放版

---

## 14.2 质量等级

- `A`: 可发布完整播放版
- `B`: 可发布，但部分页无动作或无音频
- `C`: 仅静态 deck，不开放 autoplay
- `D`: 不生成 deck，对用户隐藏播放入口
